## 0. 结论

**可行、可落地，且与当前交付边界高度吻合。** 方向定为：

> **开发方维护的「静态 3D 展包播放器 + 隐藏式编辑器」**
> - 校方/老师：只维护校内图文内容，不接触 GLB / 材质 / 热点 / VR 配置。
> - 开发方：通过静态展包 + `config.json` + 隐藏编辑面板，完成模型、全景、材质、热点、灯光、相机调优。
> - 终端用户：只看到沉浸式鉴赏界面，默认仅「自动旋转」「热点显隐」两个按钮。

该路线与现状一致：小程序工艺品模块**仅多角度图片**，后端 `craft` API **不返回 3D 字段**，VR 展馆（`hall.vr_url`）与 GLB 展包是**两条独立链路**（见 `test/README.md`）。

### 1.2 建议补强（初版未覆盖或点到未展开）
1. **坐标系与单位陷阱**：`config.model.rotation` 用**角度**需转弧度；统一 **Y-up**；热点存**模型局部坐标**（正确）——本文 7 节补充「渲染时局部→世界→屏幕」与**背面/遮挡淡化**算法。
2. **色彩管理**：全景贴图 `colorSpace = SRGBColorSpace`、`renderer.outputColorSpace = SRGBColorSpace`；GLTF 贴图 colorSpace 由 loader 处理；**慎用 emissive 提亮**（会让釉面"假自发光"）——给出调参优先级链。
3. **双语 i18n**：标题、简介、热点文案支持**中/英**（对齐现有文创详情的 zh/en 切换）。
4. **无障碍 & 动效敏感**：`prefers-reduced-motion` 时默认关自动旋转；键盘可达、ARIA、文字对比度达标。
5. **无 WebGL 降级出口**：不支持时除 poster 提示外，提供**"返回图文鉴赏"**出口（回落到多角度图片/海报）。
6. **资源压缩进阶**：`Draco` / `Meshopt` 几何压缩 + `KTX2/BasisU` 贴图压缩（decoder 随 `vendor/` 自托管），大模型可再降 50%+ 体积。
7. **播放器自身版本化**：不仅资源要 cache-busting，**通用 player 升级**也要能灰度/回滚（`viewer/v1/`、`viewer/v2/`）。
8. **config JSON Schema 校验**：提供 schema，编辑器导出即校验，杜绝手改出错。
9. **可选轻量埋点**：静态包用 `navigator.sendBeacon` 打点到既有 stats 接口（同源或已配 CORS 前提），量化"浏览/热点点击"，**不引入后台写耦合**。
10. **托管安全**：`vendor/` 自托管 Three.js（去 jsDelivr）；**编辑版与观看版物理分离**（编辑版 Basic Auth / 仅内网），生产观看版不含编辑代码。
11. **设备/浏览器测试矩阵**：iOS WKWebView、安卓微信 X5、低端机、展馆大屏；明确 WebGL2 缺失时的回退。

## 2. 总体架构

### 2.1 产品形态
一个**通用纯前端 3D 展包播放器**；每件工艺品一个静态目录：
```
https://shuyuan.gzcpu.edu.cn/exhibits/craft-001/
```
同一 URL 可被：VR 展馆热点外链 / 小程序 web-view / PC 浏览器 / 展馆大屏 / 扫码 打开。

### 2.2 观看版 vs 编辑版（物理分离，推荐）
```
viewer/            # 通用观看播放器（用户）：仅观看能力，无编辑代码
viewer-edit/       # 开发方编辑器：完整编辑面板，Basic Auth / 仅内网
exhibits/
  craft-001/config.json + assets/
  craft-002/config.json + assets/
```
> 相比 `?mode=edit` 单包切换，**分离两个包**更干净：生产包不含编辑器代码，避免被查看/误触，也利于加访问控制。`?mode=edit` 可作开发期便捷入口保留。

### 2.3 播放器模块划分（即使单文件也按此逻辑组织）
```
ViewerApp
├── ConfigLoader          # 拉取 + 校验 config.json（JSON Schema）
├── AssetLoader           # 分阶段加载 poster→GLB→panorama
├── SceneManager          # scene / camera / renderer / resize
├── ModelManager          # 加载、归一化(居中/缩放)、transform
├── MaterialManager       # 全局 + 按名覆盖 PBR
├── IBLManager            # equirect 全景 → PMREM；fallback RoomEnv
├── LightManager          # 环境/主/补/轮廓光
├── CameraController      # OrbitControls + 机位/聚焦动画
├── HotspotManager        # 局部坐标锚点、遮挡淡化、气泡/抽屉
├── ViewToolbar           # 用户极简工具条
├── EditorPanel           # 隐藏编辑面板（仅编辑版）
├── ConfigExporter        # 导出/校验/部署清单/截图 poster
├── PerfMonitor           # FPS/drawcall/体积/阈值告警
└── I18n / A11y           # 双语 + 无障碍
```
> 复杂度上来后建议迁 Vite + TypeScript（结构见初版 3.2）。第一阶段单 HTML 亦可，但**Three.js 走 `vendor/` 自托管、不依赖 jsDelivr**。

## 6. 真实全景 IBL（视觉提升最大项）

```js
async function applyPanoramaIBL(url) {
  const tex = await new THREE.TextureLoader().loadAsync(url)
  tex.mapping = THREE.EquirectangularReflectionMapping
  tex.colorSpace = THREE.SRGBColorSpace           // ← 补：色彩空间正确，避免偏色
  const pmrem = new THREE.PMREMGenerator(renderer)
  scene.environment = pmrem.fromEquirectangular(tex).texture
  if (config.environment.visibleBackground) scene.background = tex
  pmrem.dispose()
}
```
- **全景图规格**：等距柱状 2:1、sRGB、JPG；**IBL 用 ≤1.5MB（2048×1024）**，可见背景 ≤3MB（4096×2048）。`BJ.jpg` 仅作源图，**生产必须压缩分档**。
- **渐进加载**：poster → JS → GLB → 首屏出模型 → 后台加载全景 → 生成 PMREM → 平滑切到真 IBL。**大图不阻塞首屏**。
- **降级**：全景失败→RoomEnvironment；PMREM 失败→基础三点光；移动端内存紧→只用低清 IBL、不显背景；无 WebGL→poster+图文出口。
- **调参优先级（补强，避免失真）**：`toneMappingExposure` → `envMapIntensity` → 主/补光 → **最后**才少量 emissive。切忌用 emissive 硬提亮导致釉面"自发光"。

## 7. 热点系统

### 7.1 局部坐标（唯一正确做法）
打点：`Raycaster` 命中模型 → 世界坐标 → `modelGroup.worldToLocal()` → 存 `config`。
渲染：`local → modelGroup.localToWorld() → project(camera) → 屏幕像素`。这样模型缩放/居中/旋转后热点仍贴合。

### 7.2 背面 / 遮挡淡化（初版列为需求但未给实现，补上）
每帧对每个热点：
1. **背面淡化**：`dot(worldNormal, cameraDir) < 0` → 该点朝背面，`opacity` 降到 ~0.15 或隐藏。
2. **被模型遮挡**：从相机向热点世界坐标发一条 `Raycaster`，若首个命中距离 < 热点距离（中间隔着模型）→ 判定遮挡，弱化/隐藏。
> 遮挡检测每帧对 3–8 个点做射线，开销可忽略；点多时可降频（每 2–3 帧一次）。

### 7.3 讲解与相机联动
点击热点：停自动旋转 → 高亮该点、弱化其余 → 弹卡片/抽屉（双语）→ 可选 `cameraFocus` 平滑推近（tween target+distance，800ms）。点击空白关闭。

### 7.4 信息架构（补强）
卡片层级：标题 → 正文（2–4 行）→ 可选缩略图 → 可选「查看细节」。**单模型热点建议 3–8 个，上限 15**，过多则画面杂乱、移动端难点中。

## 8. 资源与加载优化

| 资源 | 建议 | 进阶（补强） |
|---|---|---|
| GLB 几何 | 普通 ≤3MB、重点 ≤5MB | `Draco`/`Meshopt` 压缩，decoder 放 `vendor/` |
| GLB 贴图 | 2048 JPG q85（精细 4096 q90） | `KTX2/BasisU` GPU 压缩，显存与带宽双降 |
| 全景 IBL | ≤1.5MB / 2048×1024 | 按设备分档：手机 2048、PC/大屏 4096 |
| 首屏总量 | ≤5MB，4G 下 3–5s | poster 秒开 + 模型淡入 |
| 第三方库 | **自托管 `vendor/`** | 去 jsDelivr，防内网/政企网拦截 |
| 缓存 | 路径带版本 `/v1/` 或资源带 hash | **player 自身**也版本化，便于灰度/回滚 |

## 9. 性能与设备矩阵（补强）

**阈值**：移动端 DPR 1.5 / PC 2；draw calls、FPS、体积超阈值编辑器告警。**不上** bloom/SSAO/实时阴影/复杂粒子/多模型同屏（移动端扛不住）。

**测试矩阵（新增，验收必过）**：
| 环境 | 关注点 |
|---|---|
| iOS Safari / 微信 WKWebView | WebGL2 支持、内存（大 GLB+全景易 OOM 白屏）、手势 |
| 安卓 微信 X5 / 系统 WebView | 兼容性、低端机 FPS、DPR 降级 |
| 中低端安卓（2–3 年前机型） | 首屏时间、发热、是否需自动降清 |
| 展馆大屏 / PC Chrome | 高清全景、全屏、侧栏讲解 |
| 弱网 4G | 渐进加载是否不阻塞、失败降级 |

## 10. 部署 · 运维 · 安全

- **目录**：`exhibits/craft-XXX/{index.html, config.json, assets/, vendor/}`（`index.html`/`vendor` 可软链到共用 player 版本）。
- **VR 外链**：VR 平台热点 → `https://…/exhibits/craft-001/`（模板 `scripts/vr-craft-hotlink.example.txt`）。
- **小程序**：仅作入口；若小程序也直接 web-view 打开单件展包，需把静态站域名加入**业务域名白名单**（`test/README.md` / 验收项）。经 VR 平台间接打开则走 VR 平台域名。
- **安全头（补强）**：Nginx 配 `Cache-Control`（见 `nginx-craft-viewer.conf.example`）、可加 `Content-Security-Policy` 限制脚本来源（自托管后可收紧）。
- **可选轻量埋点（补强）**：`navigator.sendBeacon('/api/v1/stat/craft-viewer', {id, event})` 记录浏览/热点点击；**同源或已配 CORS** 才生效，失败静默，不阻塞、不写耦合。

## 12. 风险与边界（沿用初版，补充）

- 纯前端**不能**：写服务器文件、真在线保存、多人协作、审批/审计、防查看前端代码、做真权限。→ 用"编辑导出 + 开发方上传"闭环规避。
- **不是** WeChat/小程序问题的：全景压缩、Draco、色彩管理、内存预算——都是通用 Web 3D 工程问题。
- 唯一**外部依赖**：VR 平台热点是否支持自定义外链（决定 VR→展包链路是否成立，需向 VR 厂商确认）；不成立则走「小程序文创列表 → 展包」备用入口。

## 13. 分阶段落地（A→E）

| 阶段 | 目标 | 交付 | 验收要点 |

## 14. 本模块是否涉及 UI / 视觉设计？——**是，且是核心**

这个模块**恰恰是全项目里 UI/视觉权重最高的一块**，包含两套截然不同的界面：

1. **用户侧沉浸界面（视觉决胜）**：暗场沉浸背景、ACES 影调、真实 IBL 釉面高光、poster→模型淡入的加载编排、脉冲/发光热点标记、讲解卡片（PC 侧栏）/底部抽屉（移动）、极简毛玻璃工具条、全屏/大屏模式、动效节奏与"高级感"。**做得好不好，肉眼一眼可辨。**
2. **开发侧编辑器界面（专业工具 UI）**：分组折叠面板、滑杆/数值联动、材质列表、热点列表与打点交互、预设切换、性能仪表、导出与部署清单——是一套"轻量 Web 3D 编辑器"的信息架构与交互设计。

所以它不只是"接口对接"，而是**视觉设计 + 交互设计 + 3D 观感调优**三者叠加，UI 分量很重。
