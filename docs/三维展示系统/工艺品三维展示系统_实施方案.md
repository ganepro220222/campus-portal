# 工艺品三维展示系统 · 实施方案

**项目**：云端书院小程序（贵州交通职业大学中华文化书院）
**模块**：非遗工艺品三维展示 | **版本**：V1.3（脚本边界修订）

---

## 〇、术语约定

| 术语 | 含义 |
|------|------|
| **GLB** | glTF 2.0 的二进制封装格式。单文件，几何、材质、贴图全部内嵌，无外部依赖。三维交付的标准格式。 |
| **PBR** | 基于物理的渲染。材质由「粗糙度」「金属度」等物理参数描述，而非手工调色。 |
| **IBL** | 基于图像的光照。用一张全景图作为场景的光照与反射来源，物体表面会反射出这张图的内容。**"身临其境"效果的技术本体。** |
| **等距柱状投影** | Equirectangular。宽高比严格 2:1 的全景图格式，IBL 唯一接受的输入格式。 |
| **热点（Hotspot）** | 附着在模型表面某一坐标点的可点击标记，点击弹出图文讲解。 |
| **归一化** | 将不同来源、不同尺度的模型统一缩放，并按选定策略对齐原点（**中心归零**或**底面贴地**）。 |
| **XR-FRAME** | 微信小程序官方三维框架，可作为组件内联在小程序页面中。 |
| **Three.js** | Web 端主流三维引擎，运行在 H5 页面中，能力远强于 XR-FRAME。 |

## 一、需求精确化重述

原始需求经技术梳理后重新表述为四项。**后续开发以本节为准。**

### 需求 A：三维资产自主托管

**现状**：工艺品三维模型依赖第三方平台。将 obj/mtl/jpg 上传至第三方在线服务，由其生成含播放器的离线包（约 43MB），部署后得到一个 URL。资产与展示能力均受制于该平台。

**目标**：全部模型迁移至自有 OSS，由自有服务提供展示页面，彻底解除对第三方在线服务的依赖。

**验收**：第三方平台下线后，所有三维展示功能不受影响。

### 需求 B：三维资产生产管线自主化

**现状**：每新增一件工艺品，需上传至第三方后端处理，等待返回离线包。流程不可控、不可批量、不可离线。

**目标**：建立本地处理管线，支持三类输入的批量处理，无需任何外部服务：

1. 既有第三方离线包（zip）→ 提取其中现成的 GLB
2. 现成 GLB → 校验后直接使用
3. OBJ + MTL + 贴图 → 本地转换为自包含 GLB

> **格式说明**：脚本支持 `.zip` / `.glb` / `.obj`。**FBX 不在脚本支持范围内**，需先用 Blender 等工具导出为 GLB 或 OBJ。
>
> **质量边界**：OBJ 管线主要处理**基础漫反射贴图**（`material.image`）。含 normal/roughness/metalness/AO 等多张贴图的**复杂 PBR 资产，请优先直接交付 GLB**（Blender 导出），再用批处理校验与归一化；勿指望 OBJ 转换保留完整 PBR 质感。

处理过程同时完成**贴图压缩（可配置）**与**归一化参数计算**，输出可直接入库的清单。

**验收**：一条命令处理上百个模型，输出 GLB 与 manifest.csv，全程无网络请求。

### 需求 C：三维展示能力升级

**现状**：第三方离线播放器仅支持自动旋转，无法调整光照，无法交互，无法标注。

**目标**：自建展示页，实现四项能力：

| 能力 | 精确定义 |
|------|---------|
| **C1 环境光照切换** | 用户可在多个预设环境间切换（展馆实景 / 影棚布光 / 晴天 / 阴天）。切换后模型的受光方向、高光形态、明暗对比随之改变。 |
| **C2 环境反射（核心）** | 模型表面按 PBR 规则反射当前环境。**当环境贴图使用真实展馆全景图时，工艺品釉面上会呈现展馆的射灯、展柜、墙面等实际细节**——即用户所述"身临其境"的效果。 |
| **C3 交互控制** | 拖动任意角度旋转、滚轮/双指缩放、自动旋转开关、视角重置，带阻尼惯性。 |
| **C4 热点标注** | 模型表面可预置若干标记点。**默认隐藏**，用户主动开启后才显示，保证展示画面纯净。点击标记弹出该部位的图文讲解；标记随模型旋转，转至背面时自动淡化。 |

**验收**：切换环境时，模型表面反射内容随之改变，肉眼可辨。

### 需求 D：展馆 → 工艺品的跳转链路

**现状**：小程序展馆入口经 `web-view` 打开第三方 VR 展馆。展馆内工艺品热点当前指向第三方生成的离线包 URL。

**目标**：将 VR 展馆内工艺品热点的跳转目标，改为指向自有域名下的三维展示页。

```
小程序 → web-view → 第三方 VR 展馆 → 点击工艺品热点
       → https://shuyuan.gzcpu.edu.cn/craft/{id}   ← 自有服务
       → 从自有 OSS 加载 GLB + 全景图
```

**验收**：从展馆点击工艺品，进入自有展示页，功能完整。

## 三、总体架构

### 3.1 双入口设计（重要）

> **微信硬约束**：小程序中的 `<web-view>` 组件**独占整个页面**，无法与按钮、文字、价格等元素并存于同一屏。

因此**必须**保留两套展示实现，各司其职，**共用同一份 GLB 资产**：

| 入口 | 场景 | 技术实现 | 能力 |
|------|------|---------|------|
| **入口 1：商品详情页** | 同屏显示【模型 + 商品名 + 联系购买】 | **XR-FRAME 内联组件**（已存在，保留并扩展 transform） | 轻量预览：自转、缩放 |
| **入口 2：沉浸式鉴赏页** | 全屏无干扰的深度鉴赏 | **H5 + Three.js**（新建） | 完整能力：C1~C4 |

**连接方式**：

- 商品详情页底部增加「沉浸式鉴赏」按钮 → 专用 web-view 页 → H5 `/craft/{id}`
- VR 展馆的工艺品热点 → 直接打开 H5 `/craft/{id}`

### 3.2 H5 鉴赏页部署设计（实施前必须明确）

当前仓库有 `admin/`、`miniapp/`、`backend/`，**没有** H5 viewer 工程。推荐：

```
viewer/                    ← 新建轻量前端包（Vite + Three.js，无 Element Plus）
  src/
  dist/                    ← 构建产物

部署：
  Nginx / CDN 静态目录
  /craft/*  → viewer/index.html（SPA fallback）
  /api/*    → 反向代理到 backend:8080

页面加载：
  GET /api/v1/crafts/{id}/viewer  → 拉取 modelUrl / transform / envPresets ...
  Three.js fetch GLB + 全景图（OSS 公共读 + CORS）
```

**不要**把 H5 viewer 塞进 `admin` 构建——管理端与公开展示页发布节奏、缓存策略不同。

### 3.3 小程序 web-view 承载页设计

**不要复用** `packageC/college/webview`（学院内容语义、标题、统计会混乱）。

建议新增专用页面，例如 `packageA/craft/viewer-webview`：

| 项 | 建议 |
|---|---|
| 路由 | `/packageA/craft/viewer-webview?id={craftId}` |
| 参数 | 优先传 `id`，页面内拼 `https://shuyuan.gzcpu.edu.cn/craft/{id}`；若传 `url` 则**限制 host 为自有域名** |
| 标题 | 「立体鉴赏」或工艺品名称 |
| 显示条件 | 后端 `viewerEnabled=true` 且鉴赏 URL 可访问 |
| 失败降级 | Toast + 返回详情页；可选提示复制链接到浏览器 |

`app.json` 需在 `packageA` 分包注册该页面。

## 四、关键技术约束（开发前必读）

### 约束 1：OSS 必须开启 CORS

Three.js 通过 `fetch` 加载 GLB 与全景图，属跨域请求。OSS 若未配置 CORS，浏览器直接拦截，模型加载失败。

```
来源(Origin)  : https://shuyuan.gzcpu.edu.cn
允许 Methods  : GET, HEAD
允许 Headers  : *
暴露 Headers  : ETag, Content-Length
缓存时间      : 86400
```

### 约束 2：三维资产用公共读 + CDN，不要用短签名 URL

签名 URL 带时效参数，会导致 CDN 与浏览器缓存失效。

三维模型属公开展示内容，无隐私风险。**推荐**：`models/` 前缀公共读 + CDN 长缓存（`Cache-Control: max-age=31536000`），**文件名带内容哈希**。

> **与现网差异**：当前 `OssService` 对多数媒体走私有桶 + 签名 URL；管理端 `model3d` 上传已返回 `buildPublicUrl()` 固定 CDN 地址。实施时需：**①** 模型/全景图走公共读前缀；**②** 新增 `/admin/crafts/{id}/model` 时做 GLB 校验 + 哈希命名；**③** 不要对 3D 公开展示资源套用视频那种短时效签名。

### 约束 3：热点坐标必须存「模型局部坐标」

若存世界坐标，模型一旋转，热点立即错位。

```javascript
const local = model.worldToLocal(intersects[0].point.clone());
worldPos.copy(hotspot.localPos).applyMatrix4(model.matrixWorld);
```

### 约束 4：GLB 贴图必须内嵌

入库前解析 GLB JSON 块，确认 `images[]` 均含 `bufferView`（非 `uri` 外链）。`batch_glb.py` / `obj2glb.py` 已做基础校验；后端上传接口应复用同等逻辑。

### 约束 5：全景图必须是等距柱状投影（2:1）

IBL 只接受宽高比严格 2:1 的 equirectangular 图。VR 平台内部的立方体六面图/瓦片图**不能直接用**。

**资产规格（验收前置）**：

| 项 | 建议 |
|---|---|
| 分辨率 | 2048×1024 或 4096×2048 |
| 格式 | JPEG（质量 85）或 WebP；色彩空间 sRGB |
| 注意 | 避免过曝；每展厅至少 1 张 |
| 备选 | 若只有 cube map，需增加 cube→equirectangular 转换步骤 |
| 验收样例 | 用一件高反射釉面陶瓷对比「影棚预设 vs 展馆实景」 |

若无法获取真实展馆全景图，C2 退化为通用预设环境（方案仍可做，效果打折）。

### 约束 6：`shuyuan.gzcpu.edu.cn` 必须加入微信「业务域名」

小程序 `web-view` 只能打开业务域名白名单内的页面。已列入《上线分工 Checklist》§⑤。

### 约束 7：模型必须在入库时完成归一化（策略二选一）

不同来源 GLB 尺度与原点差异极大。归一化参数**上传/批处理时计算一次**存库，前端套用。

**本方案默认策略：几何中心归零**（适合绕中心旋转）：

```json
{ "scale": 1.35, "offsetX": 0, "offsetY": -0.08, "offsetZ": 0 }
```

批处理脚本同时输出 `floorOffsetY`（底面贴地，适合展台模式）。H5/XR-FRAME 实施前**选定一种**，不要混用。

## 五、数据模型设计

### 5.0 分阶段：MVP 与增强版

避免一次性上全表。结合当前 `craft` 仅 `model_3d_url` + `preview_type` 的现状：

**MVP（先让沉浸式鉴赏页上线）**

| API | `GET /crafts/{id}/viewer`、`PUT /admin/crafts/{id}/viewer-config`、`POST /admin/crafts/{id}/model` |

```
GET /api/v1/crafts/{id}/viewer
```

响应体含 `modelUrl`（来自 `model_3d_url`）、`transform`、`camera`、`material`、`posterUrl`、`viewerEnabled`；增强版再含 `hotspots`、`envPresets`。

### 6.2 管理端接口

**MVP**

```
POST   /api/v1/admin/crafts/{id}/model          上传 GLB → 校验 → 哈希命名 → 写 model_3d_url + transform_json
PUT    /api/v1/admin/crafts/{id}/viewer-config  保存 poster / camera / material / viewer_enabled
```

**增强版**

```
GET/POST/PUT/DELETE  .../hotspots、.../env-presets  （见原完整 CRUD 设计）
```

### 6.3 与现网上传链路的衔接

| 现状 | 目标 |
|---|---|
| `OssUploadInput` + `/admin/upload?scene=model3d` | 保留为运营同学习惯 |
| 仅扩展名校验 | 新增 `/admin/crafts/{id}/model`：**GLB magic、内嵌贴图、大小上限、Draco 等扩展 warning、SHA 命名** |

## 七、分步实施计划

### 阶段 1：资产处理管线（1 天，可立即开始）

```bash
cd docs/三维展示系统
pip install trimesh pillow numpy

# 递归扫描；输出目录在输入树下会自动排除；输出名 = {路径}-{glbSha1前8}.glb
python batch_glb.py 工艺品源文件 -o 处理结果 --max-texture 2048

# 单文件；默认输出 模型名-{glbSha1前8}.glb，sidecar 含 sourceSha1/objBundleSha1/glbSha1
python obj2glb.py 某模型/model.obj --max-texture 2048 --sidecar
```

检查 `manifest.csv`：

- `glbSha1` 变化 → CDN 换名即失效旧缓存
- `objBundleSha1`（仅 OBJ）→ MTL/贴图变更可追溯，即使 .obj 文件未改
- 「贴图内嵌」为否 → 需修复
- 备注含 `有alpha改PNG` / `JPEG丢弃alpha` → 核对透明区域

**贴图策略**：

| 场景 | 建议 |
|---|---|
| 默认 | `--texture-format auto`（有透明→PNG，否则 JPEG） |
| 强制 JPEG 且含透明 | 默认改 PNG；确需丢 alpha 加 `--allow-alpha-loss` |
| 已是 PBR GLB | 直接 `.glb` 输入（原样复制，不压缩贴图） |
| OBJ 仅缩放不重编码 | `--no-reencode`（**仅 OBJ 分支生效**） |

**输出目录**：必须使用**独立目录**（`-o` 不能与输入目录相同）；若 `-o` 为输入子目录（如 `source/output_glb`），脚本自动排除该子目录避免重复处理。

**源资产命名**：路径建议英文/数字/下划线；贴图路径含空格时请用 MTL 引号包裹；复杂 PBR 仍优先 GLB 直传。

**交付物**：`*.glb` + `manifest.csv`（`sourceSha1`、`glbSha1`、`objBundleSha1` 相对路径内容哈希、transform 字段）

### 阶段 2：后端 MVP（1.5 天）

## 十、注意事项与风险

### 10.1 性能

| 项 | 建议 |
|----|------|
| 单个 GLB | **≤ 3MB**；贴图 2048 + JPEG 通常足够 |
| 全景图 | **≤ 1.5MB**（2048×1024 JPEG q=85） |
| 缓存 | 文件名带内容哈希，更新换名 |
| 移动端 | `setPixelRatio(min(dpr, 2))` |

### 10.2 质感调校经验值

（陶瓷釉面 roughness 0.10~0.20、metalness ~0、envMapIntensity 1.2~1.8 等——见原表，略）

### 10.3 热点工作量

10~20 件重点展品配置热点即可，其余纯 3D 旋转+环境反射。

### 10.4 开发规范

高内聚低耦合；注释中文；生产代码与文档不出现第三方平台名称。

### 10.5 降级策略

WebGL 不可用 → poster；GLB 失败 → 重试；全景失败 → 内置影棚环境；无热点 → 隐藏开关。

### 10.6 资产管线风险（脚本）

| 风险 | 缓解（V1.3） |
|---|---|
| OBJ 只 hash .obj 导致 MTL/贴图变更不换名 | 输出文件名以 **`glbSha1`** 为准；`objBundleSha1` 追溯 MTL/贴图 |
| 输出目录在输入树下被重复扫描 | 真子目录自动排除；**输入=输出直接报错** |
| `--texture-format jpeg` 丢 alpha | 默认改 PNG；`--allow-alpha-loss` 才允许丢弃 |
| `compute_transform` 返回 None 致 TypeError | 统一抛 `RuntimeError`，批处理标记归一化失败不中断 |
| `objBundleSha1` 含绝对路径 | 改为**相对 obj 目录路径 + 文件内容** hash |
| MTL 贴图路径含空格 | 支持引号路径；无引号路径建议避免空格 |
| OBJ 管线 PBR 能力有限 | 复杂 PBR 优先 GLB 直传 |
