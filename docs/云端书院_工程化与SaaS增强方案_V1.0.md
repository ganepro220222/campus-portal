## 一、背景与目标

### 1.1 背景

当前项目（截至 V1.2 实施进度）已具备：

- 小程序主要页面 UI 与核心浏览/报名/个人中心链路；
- 管理后台 9 类内容 CRUD + 品牌化壳层；
- Spring Boot 分层架构、JWT/RBAC、活动报名事务与 Docker 开发环境。

在**功能清单**逐步补全的前提下，若缺少工程化配套，项目综合质量约 **8.0–8.5/10**（满足合同功能验收与试运行），但在**可重复发布、故障发现、性能基线、长期运维**方面仍明显弱于商业 SaaS。

### 1.2 增强目标

| 维度 | 当前（约） | 增强后目标 |
|------|------------|------------|
| 综合质量（对内） | 7.5/10 | **8.5–9.0/10** |
| 功能完整度（125 项） | 65–70% | **95–100%** |
| 自动化测试 | 极少量 | 核心路径可回归 |
| 生产就绪 | 低 | 可正式上线试运行 |
| 故障发现/定位 | 依赖人工 | 监控 + 日志 + 告警 |
| 安全与审计 | 基础 | 满足验收清单 + 可追责 |

### 1.3 四条原则（增强包核心）

1. **测得住** — 自动化测试 + CI，发布可回归  
2. **看得见** — 日志、指标、告警，故障可发现可定位  
3. **扛得住** — 压测基线、缓存、备份、回滚、降级  
4. **管得住** — 安全加固、操作审计、文档与发布流程

## 二、现状基线与目标差距

### 2.1 已具备（不必重复建设）

| 项 | 说明 |
|----|------|
| 分层架构 | Controller / Service / Mapper，api/admin 双前缀 |
| 统一响应与异常 | `Result`、`BusinessException`、`GlobalExceptionHandler` |
| 鉴权 | 小程序 JWT、Admin RBAC、登录锁定（Redis） |
| 核心业务 | 活动报名事务、名额控制、Admin 审核导出 |
| 开发环境 | Docker Compose（MySQL + Redis + Backend） |
| Admin 构建 | Vite 生产构建已通过 |
| 搜索索引 | `search_index` 与 Admin 变更同步机制 |

### 2.2 主要差距（增强包需补齐）

| 类别 | 现状 | SaaS 级要求 |
|------|------|-------------|
| 自动化测试 | 2 个基础测试 | 核心 Service 单测 + 关键 API 集成测试 |
| CI/CD | 手动构建部署 | 推送即构建、测试、镜像、可部署 artifact |
| 监控告警 | 无 | 指标 + 日志 + 告警通道 |
| 生产配置 | 仅 dev/docker | prod/staging 环境分离、密钥外置 |
| 审计日志 | 表结构可能有，未系统落地 | Admin 写操作全量可追溯 |
| 压测报告 | 无 | 对照验收标准 P95 指标 |
| Mock 兜底 | 小程序大量 mock fallback | 生产环境失败明确提示，非静默假数据 |
| 部分 API | 反馈、积分写入、EventLog 埋点等缺失 | 业务闭环完整 |
| 运维文档 | 分散 | 部署/运维/管理员手册齐全 |

## 四、工程与质量保障

### 4.1 后端自动化测试

**目标：** 核心逻辑变更可快速回归，避免报名/权限/积分等回归事故。

| 任务 | 说明 | 优先级 |
|------|------|--------|
| Service 单测 | `EnrollService`、`AuthService`、`ProfileService`、`Admin*Service` 核心方法 | P0 |
| Controller 集成测试 | `@WebMvcTest` 或 `@SpringBootTest` + MockMvc：登录、报名、403/401 | P0 |
| 测试数据 | H2 或 Testcontainers MySQL；每测独立事务回滚 | P1 |
| 覆盖率门禁 | JaCoCo，核心 module **≥60%**（逐步达成，先覆盖 P0 类） | P1 |

**建议目录：**

```
backend/src/test/java/com/shuyuan/backend/
├── service/EnrollServiceTest.java
├── service/AuthServiceTest.java
├── controller/ActivityControllerIT.java
└── ...
```

### 4.2 前端与管理后台

| 任务 | 说明 | 优先级 |
|------|------|--------|
| Admin lint | ESLint + vue-tsc 纳入 CI | P1 |
| 关键页面 smoke | 登录 → Dashboard → 新闻列表加载 | P2 |
| 小程序 | 微信开发者工具 CLI 或 miniprogram-ci 上传体验版（可选） | P2 |

### 4.3 CI/CD（GitHub Actions 示例）

**流水线阶段：**

1. **lint** — Admin `npm run build`；Backend 编译  
2. **test** — `mvn test`；失败阻断合并  
3. **build** — Docker 镜像 build；Admin `dist` 产物  
4. **artifact** — 推送镜像至 Registry（需校方/运维提供仓库，或乙方私有 Registry）  
5. **deploy** — staging 自动部署；prod 手动审批（推荐）

**分支策略：**

- `main` — 生产就绪，保护分支，仅 PR 合并  
- `develop` 或 `staging` — 集成测试环境  

### 4.4 API 契约与回归

| 任务 | 说明 |
|------|------|
| OpenAPI / Swagger | SpringDoc 生成 `/v3/api-docs`，与实现同步 |
| Apifox 集合 | 导出环境变量（dev/staging/prod）；核心接口回归集 |
| 变更纪律 | 接口 breaking change 必须更新文档与 Apifox |

## 五、可观测性与运维

### 5.1 结构化日志

| 项 | 规范 |
|----|------|
| 格式 | JSON 或统一 pattern：`traceId`、`memberId/adminId`、`uri`、`durationMs` |
| 级别 | 生产默认 INFO；DEBUG 仅 dev |
| 敏感信息 | 禁止打印 password、token 全文、身份证号 |
| 关联 | 网关或 Filter 生成 `X-Trace-Id`，全链路透传 |

**实现建议：** Logback JSON Encoder + MDC；或接入云日志（需校方云账号）。

### 5.2 指标监控

| 指标 | 说明 |
|------|------|
| `http_server_requests` | QPS、P50/P95/P99、按 uri 分组 |
| JVM | 堆内存、GC、线程 |
| MySQL | 连接池活跃数、慢查询计数 |
| Redis | 命中率、连接数 |
| 业务 | DAU、报名次数、AI 调用次数（可选） |

**技术选型（二选一）：**

- 轻量：Spring Boot Actuator + Prometheus + Grafana（自建或 Docker）  
- 云服务：阿里云 ARMS / 腾讯云监控（需校方账号）

### 5.3 告警规则（最低集）

| 告警 | 条件 | 通道 |
|------|------|------|
| 接口错误率 | 5xx > 1% 持续 5min | 钉钉/企微 Webhook |
| 接口延迟 | 核心 API P95 > 500ms 持续 10min | 同上 |
| 服务不可用 | 健康检查失败 | 同上 |
| 磁盘/内存 | > 85% | 同上 |

### 5.4 健康检查扩展

现有 `HealthController` 扩展为：

```
GET /api/v1/health
→ { "status": "UP", "db": "UP", "redis": "UP", "version": "x.y.z" }
```

Docker / Nginx 探活依赖此接口。

### 5.5 运行手册（Runbook）

文档化以下内容（见第十章）：

- 日常巡检项  
- 告警收到后第一步操作  
- 回滚步骤  
- 备份恢复演练步骤（每季度至少 1 次）

## 六、安全加固

> 实现责任在乙方；证书、域名、WAF 等资源需校方配合。

### 6.1 传输与接入

| 项 | 说明 |
|----|------|
| 全站 HTTPS | Nginx 终止 SSL；HTTP 301 跳转 |
| 安全响应头 | `X-Content-Type-Options`、`X-Frame-Options`、`Referrer-Policy` |
| CORS | Admin 生产域名白名单；禁止 `*` |

### 6.2 鉴权与限流

| 项 | 说明 |
|----|------|
| Token 策略 | 过期时间、刷新策略文档化；Admin 会话超时 |
| 接口限流 | 登录、AI 问答、报名等热点接口 Redis 滑动窗口 |
| 防暴力 | 登录锁定（已有）；扩展至敏感接口 |

### 6.3 输入与文件

| 项 | 说明 |
|----|------|
| 参数校验 | `@Valid` 全覆盖；自定义业务校验在 Service |
| 上传白名单 | 扩展名 + MIME + 大小；拒绝可执行文件 |
| OSS 私有读 | 签名 URL；直链 403（验收清单要求） |

### 6.4 密钥管理

| 项 | 说明 |
|----|------|
| 禁止入库 | `.env`、`application-prod.yml` 敏感项仅环境变量 |
| `.gitignore` | 确认 `.env*`、密钥文件已忽略 |
| 轮换 | 文档说明 JWT secret、DB 密码轮换流程 |

### 6.5 操作审计日志

**目标：** Admin 所有写操作可追溯。

| 字段 | 说明 |
|------|------|
| admin_id | 操作人 |
| action | create/update/delete/publish/export |
| resource | news/hall/activity/... |
| resource_id | 目标 ID |
| ip | 请求 IP |
| created_at | 时间 |
| detail | JSON 摘要（变更前后可选） |

**实现：** AOP 拦截 `@PostMapping/@PutMapping/@DeleteMapping` on admin controllers，或 Service 层统一记录。

### 6.6 AI 与内容安全

| 项 | 说明 |
|----|------|
| 输入审核 | 问答前文本审核（需 AI 平台或第三方 API，Key 校方提供） |
| 输出审核 | 生成内容返回前审核 |
| 限次 | 每日上限（已有设计，需完整实现 + Redis） |

## 七、性能与容量

### 7.1 压测基线（对照交付物 §五）

| 指标 | 目标 | 工具 |
|------|------|------|
| 热点接口 P95（缓存命中） | < 50ms | ab / wrk |
| 冷查询 P95 | < 300ms | ab |
| 200 并发浏览 | P95 < 500ms | JMeter |
| 50 并发报名 | 名额不超卖 | ab -c 50 |

**交付物：** `docs/perf/压测报告_YYYYMMDD.md`（或独立目录）

### 7.2 缓存策略

| 场景 | 策略 |
|------|------|
| 首页/Banner/公告 | Redis 缓存 + TTL（公告已有 5min） |
| 详情浏览量 | Redis INCR + 定时落库（方案已有，需完整实现） |
| 分类列表 | 长 TTL + Admin 变更清缓存 |
| 搜索 | 维持 DB + FULLTEXT；热点词可选缓存 |

### 7.3 数据库

| 项 | 说明 |
|----|------|
| 慢查询 | `slow_query_log` 开启；每周 review |
| 索引 | `enroll`、`event_log`、`search_index` 等高频查询复盘 |
| 分页 | 深分页避免 `OFFSET` 过大；游标或 id 范围 |

### 7.4 小程序

| 项 | 说明 |
|----|------|
| 分包体积 | 主包 < 2MB 建议；审计 unused 资源 |
| 图片 | CDN + 懒加载 + 合适分辨率 |
| 生产 mock | **关闭或降级** mock fallback，见 §9.1 |

## 八、架构与高可用

> 不引入 K8s/微服务，与 V1.2「极简架构」原则一致。

### 8.1 环境分离

| 环境 | 用途 | 配置 |
|------|------|------|
| dev | 本地 Docker + 开发者 | `application-dev.yml` |
| staging | 预发联调、甲方验收 | `application-staging.yml` |
| prod | 正式上线 | `application-prod.yml` |

**禁止：** prod 使用 dev 密码、调试日志、mock 数据。

### 8.2 应用无状态

- Session / Token 状态存 Redis；应用实例可水平扩展  
- 本地文件上传**禁止**（必须 OSS，需校方 Bucket）

### 8.3 数据库与 Redis

| 项 | 说明 |
|----|------|
| MySQL | 推荐云 RDS；每日自动备份；**恢复演练** |
| Redis | 云 Redis 主从；持久化策略文档化 |
| 迁移 | Flyway 或 Liquibase；禁止手工改 prod 表结构 |

### 8.4 发布与回滚

| 项 | 说明 |
|----|------|
| 版本号 | SemVer；Git tag 对应 release |
| 数据库 | 迁移脚本向前兼容；回滚脚本或备份恢复 |
| 应用 | 保留上一版 Docker 镜像；一键回滚文档 |
| 小程序 | 体验版 → 审核 → 发布；版本号记录 |

### 8.5 降级预案

| 场景 | 降级策略 |
|------|----------|
| AI 不可用 | 友好提示 + FAQ 静态兜底（可选） |
| OSS 不可用 | 只读；禁止上传 |
| Redis 不可用 | 限流/缓存失效；核心写操作仍依赖 DB |
| 搜索不可用 | 明确错误提示；**生产禁用 mock 静默兜底** |

## 九、产品体验与业务深度

### 9.1 生产环境 Mock 策略

| 环境 | 策略 |
|------|------|

**实现：** `app.js` 或 `request.js` 中 `globalData.useMock = false`（按环境变量或 build 模式）。

| 消息中心 | 小程序消息列表页 + 已读 | P1 |

## 十一、团队流程与发布规范

### 11.1 Git 流程

- `main` 保护：禁止 force push；PR 至少 1 人 Review  
- Commit message：与现有风格一致（feat/fix/chore + 中文说明）  
- Release：打 tag `v1.0.0`；附 CHANGELOG  

### 11.2 PR 检查清单（建议贴入 PR 模板）

- [ ] 关联需求/Issue  
- [ ] 本地/CI 测试通过  
- [ ] 接口变更已更新 Apifox/Swagger

## 十六、风险与不做增强的后果

| 风险 | 不做增强的可能后果 |
|------|---------------------|
| 无自动化测试 | 每次发版回归靠人工，易引入报名/权限回归 bug |
| 无监控 | 故障依赖用户反馈，夜间/假期响应慢 |
| 无压测 | 开学/活动高峰可能出现慢响应或超卖争议 |
| 无审计 | 后台误操作无法追责 |
| mock 进生产 | 用户看到假数据，严重损害信任 |
| 无回滚 | 一次坏发布导致长时间不可用 |
