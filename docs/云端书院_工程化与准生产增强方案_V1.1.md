## 一、背景与目标

### 1.1 背景

当前项目（截至 V1.2 实施进度）已具备：

- 小程序主要页面 UI 与核心浏览/报名/个人中心链路；
- 管理后台 9 类内容 CRUD + 品牌化壳层；
- Spring Boot 分层架构、JWT/RBAC、活动报名事务与 Docker 开发环境。

在**功能清单**逐步补全的前提下，若缺少工程化配套，项目综合质量约 **8.0–8.5/10**（满足合同功能验收与试运行），但在**可重复发布、故障发现、性能基线、长期运维**方面仍明显弱于准生产级交付要求。

### 1.2 增强目标

| 维度 | 当前（约） | 增强后目标 |
|------|------------|------------|
| 综合质量（对内） | 7.5/10 | **8.5–9.0/10** |
| 功能完整度（125 项） | 65–70% | **95–100%** |
| 自动化测试 | 极少量 | 核心高风险场景可回归 |
| 生产就绪 | 低 | 可正式上线试运行 |
| 故障发现/定位 | 依赖人工 | 监控 + 日志 + 告警 |
| 安全与审计 | 基础 | 满足《交付物与验收标准》§六 + 可追责 |

### 1.3 四条原则（增强包核心）

1. **测得住** — 风险场景测试 + CI，发布可回归  
2. **看得见** — 日志、指标、告警，故障可发现可定位  
3. **扛得住** — 压测基线、缓存、备份、回滚、降级  
4. **管得住** — 安全加固、操作审计、文档与发布流程  

### 1.4 实施节奏（V1.1 修订）

**工程化增强不作为功能完成后的单独补丁**，而是按优先级分批并入 Phase 0–8：

- **尽早并行**：测试、CI、环境隔离、结构化日志、Mock 关闭策略（E1 与功能开发同步启动）  
- **staging 就绪后**：压测、告警、备份恢复演练、Runbook  
- **上线前硬门禁**：见 [第十五章](#十五上线门禁清单)，不满足不建议正式发布

## 二、术语边界：本项目不做多租户 SaaS

| 概念 | 本项目含义 | **不包含** |
|------|------------|------------|
| 「准生产级 / 可运维交付线」 | 工程质量、可监控、可回滚、可持续运维 | 多租户、计费套餐、租户隔离 |
| 「增强包」 | 在 V1.2 功能 Phase 之上提高**质量下限** | 替代功能开发或引入微服务/K8s/ES |
| 「SaaS 增强」（历史用语，V1.1 已弃用） | — | 商业 SaaS 及格线、多校平台化 |

**项目定位对齐：** 《技术实施方案 V1.2》强调「极简架构、低维护」——单校书院小程序，不引入微服务、K8s、Elasticsearch。本增强包与此一致。

**将来扩展：** 若需多学校 SaaS，单独立项「多租户架构增强包」，含租户模型、数据隔离、套餐计费等，**不得与当前交付混谈**。

| 部分 API | 反馈、积分写入、EventLog 埋点等缺失 | 业务闭环完整 |

## 六、工程与质量保障

### 6.1 测试策略：风险场景导向（非覆盖率导向）

JaCoCo 核心 module ≥60% **可保留为 P1 参考指标**，但**不作为主要目标**。本项目高风险场景如下：

| 场景 | 必测内容 | 优先级 |
|------|----------|--------|
| 登录与权限 | 微信登录、学号登录、JWT 过期、Admin 越权 403 | P0 |
| 活动报名 | 防重复、名额满、**50 并发不超卖**、取消释放名额、审核占名额 | P0 |
| 文件上传 | 类型白名单、大小限制、伪装扩展名、OSS 签名 URL | P0 |
| 统计埋点 | PV/UV/DAU、浏览量 5 分钟落库、下载记录、学习足迹 | P0 |
| 积分徽章 | 每日上限、防刷、重复行为不重复加分 | P1 |
| AI 问答 | 未登录不可用、每日 20 次限制、输入/输出内容安全、无知识库时友好提示 | P1 |
| 后台审计 | 发布、下架、导出、删除、配置变更均入审计日志 | P0 |
| 生产 Mock | prod 构建检测 mock fallback，**不允许静默假数据** | P0 |

### 6.2 后端自动化测试

| 任务 | 说明 | 优先级 |
|------|------|--------|
| Service 单测 | `EnrollService`、`AuthService`、`ProfileService` 等核心方法 | P0 |
| Controller 集成测试 | 登录、报名、403/401 | P0 |
| 并发报名脚本 | 50 并发不超卖（可 JMeter / ab） | P0 |
| 测试数据 | H2 或 Testcontainers MySQL；每测独立事务回滚 | P1 |
| 覆盖率门禁 | JaCoCo ≥60%（逐步达成） | P1 |

**建议目录：**

```
backend/src/test/java/com/shuyuan/backend/
├── service/EnrollServiceTest.java
├── service/AuthServiceTest.java
├── controller/ActivityControllerIT.java
└── ...
```

### 6.3 前端与管理后台

| 任务 | 说明 | 优先级 |
|------|------|--------|
| Admin lint + build | ESLint + vue-tsc 纳入 CI | P1 |
| 关键页面 smoke | 登录 → Dashboard → 新闻列表 | P2 |
| 小程序真机 | 微信开发者工具 / miniprogram-ci 体验版 | P1（试运行期） |

### 6.4 CI/CD（GitHub Actions 示例）

**流水线阶段：**

1. **lint** — Admin `npm run build`；Backend 编译  
2. **test** — `mvn test`；失败阻断合并  
3. **build** — Docker 镜像 build；Admin `dist` 产物  
4. **artifact** — 推送镜像至 Registry  
5. **deploy** — staging 自动部署；prod **手动审批**

**分支策略：**

- `main` — 生产就绪，保护分支，仅 PR 合并  
- `develop` 或 `staging` — 集成测试环境

## 八、可观测性与运维

> **轻量默认，完整可选** — 符合「极简架构、低维护」原则。

### 8.1 低成本默认版（P0，上线前必须）

| 项 | 说明 |
|----|------|
| Spring Boot Actuator | `/health` 含 DB、Redis、version |
| Nginx access/error log | 接入日志轮转 |
| 应用 JSON 日志 + traceId | Logback + MDC |
| 云服务器告警 | CPU/内存/磁盘 > 85% |
| 5xx 错误率告警 | > 1% 持续 5min → 钉钉/企微 Webhook |
| 数据库备份失败告警 | 备份任务失败即告警 |

**健康检查扩展：**

```
GET /api/v1/health
→ { "status": "UP", "db": "UP", "redis": "UP", "version": "x.y.z" }
```

### 8.2 增强版（P1，试运行期补齐）

| 项 | 说明 |
|----|------|
| Prometheus + Grafana | 或阿里云 ARMS / 腾讯云监控 |
| Loki 或云日志 | 集中检索 |
| 慢查询可视化 | MySQL slow_query_log |
| AI 调用量/费用仪表盘 | 额度预警 |
| 业务指标看板 | DAU、报名数、AI 调用数、上传失败数 |

### 8.3 结构化日志规范

| 项 | 规范 |
|----|------|
| 格式 | JSON 或统一 pattern：`traceId`、`memberId/adminId`、`uri`、`durationMs` |
| 级别 | 生产默认 INFO；DEBUG 仅 dev |
| 敏感信息 | 禁止打印 password、token 全文、身份证号 |
| 关联 | Filter 生成 `X-Trace-Id`，全链路透传 |

### 8.4 Runbook 与 RPO/RTO

| 指标 | 目标 | 说明 |
|------|------|------|
| **RPO** | ≤ 24 小时 | 最多丢失一天内数据；启用 binlog 可更低 |
| **RTO** | ≤ 4 小时 | 故障后 4 小时内恢复核心访问 |
| 恢复演练 | 首次上线前 ≥1 次；之后每季度 ≥1 次 | 与技术方案「每日全量备份、每季度验证可恢复性」对齐 |

Runbook 文档化：日常巡检、告警第一步、回滚步骤、备份恢复步骤。

## 九、安全、隐私与内容合规

> 实现责任在乙方；对齐《交付物与验收标准 V1.2》§六。

### 9.1 传输与接入

| 项 | 说明 |
|----|------|
| 全站 HTTPS | Nginx 终止 SSL；HTTP 301 跳转 |
| 安全响应头 | `X-Content-Type-Options`、`X-Frame-Options`、`Referrer-Policy` |
| CORS | Admin 生产域名白名单 |

### 9.2 鉴权与限流

| 项 | 说明 |
|----|------|
| Token 策略 | 过期时间、刷新策略文档化 |
| 接口限流 | 登录、AI 问答、报名等 Redis 滑动窗口 |
| 防暴力 | 登录锁定（已有）；扩展至敏感接口 |

### 9.3 输入与文件

| 项 | 说明 |
|----|------|
| 参数校验 | `@Valid` 全覆盖 |
| 上传白名单 | 扩展名 + MIME + 大小；拒绝可执行文件 |
| OSS 私有读 | 签名 URL；直链 403 |

### 9.4 密钥管理

| 项 | 说明 |
|----|------|
| 禁止入库 | 敏感项仅环境变量 |
| `.gitignore` | 确认 `.env*` 已忽略 |
| 交接 | 密钥不通过微信群/邮件明文传递（见交付物 §七） |

### 9.5 操作审计日志

Admin 所有写操作可追溯：`admin_id`、`action`、`resource`、`resource_id`、`ip`、`created_at`、`detail`（JSON 摘要）。

### 9.6 AI 与内容安全

| 项 | 说明 |
|----|------|
| 输入/输出审核 | 问答前后文本审核 |
| 限次 | 每日上限 + Redis |
| 额度预警 | P1 试运行期配置 |

| 指标 | 目标值 | 测试方法 |
|------|--------|----------|
| 接口响应（缓存命中） | P95 < 50ms | ab 压测热点接口 |
| 接口响应（数据库查询） | P95 < 300ms | ab 压测冷接口 |
| 小程序首屏加载 | < 2s（4G） | 真机测试 |
| 视频首帧加载 | < 3s（4G） | 真机测试 |
| 并发浏览 | 200 并发 P95 < 500ms | JMeter |
| 并发报名 | 50 并发不超卖 | ab -c 50 / JMeter |

**交付物：** `docs/perf/压测报告_YYYYMMDD.md`

### 10.1 缓存与数据库

| 场景 | 策略 |
|------|------|
| 首页/Banner/公告 | Redis 缓存 + TTL |
| 详情浏览量 | Redis INCR + 定时落库（5 分钟） |
| 慢查询 | `slow_query_log` 开启；P1 每周 review |
| 小程序分包 | 主包 < 2MB；审计 unused 资源 |

## 十一、环境分离、发布与回滚

> 不引入 K8s/微服务，与 V1.2「极简架构」一致。

### 11.1 环境分离

| 环境 | 用途 | Mock 策略 |
|------|------|-----------|
| dev | 本地 Docker | 允许 mock fallback |
| staging | 预发联调、甲方验收 | 默认不用 mock，除非显式开启 |
| prod | 正式上线 | **禁止静默 mock** |

**禁止：** prod 使用 dev 密码、DEBUG 日志、mock 数据。

### 11.2 应用无状态与存储

- Session / Token 存 Redis；应用可水平扩展  
- 本地文件上传**禁止**（必须 OSS）

### 11.3 数据库与 Redis

| 项 | 说明 |
|----|------|
| MySQL | 云 RDS；每日自动备份；恢复演练 |
| Redis | 云 Redis；持久化策略文档化 |
| 迁移 | Flyway 或 Liquibase |

### 11.4 发布与回滚

| 项 | 说明 |
|----|------|
| 版本号 | SemVer；Git tag 对应 release |
| 应用 | 保留上一版 Docker 镜像；回滚演练记录 |
| 小程序 | 体验版 → 审核 → 发布 |

### 11.5 Git 流程与 PR 检查清单

- `main` 保护；PR Review；Release 打 tag + CHANGELOG  
- PR 模板：测试通过、Apifox 更新、无密钥、含迁移脚本

## 十二、Mock、降级与容错策略

| 环境 | 策略 |
|------|------|
| dev | 可用 mock，便于离线开发 |
| staging | 默认不用 mock；接口失败 → 空态/错误态 |
| prod | **禁止静默 mock**；失败明确 Toast/错误页 |

| 模块 | 降级策略 | 用户感知 |
|------|----------|----------|
| AI 不可用 | FAQ 静态兜底（可选） | 明确提示「AI 暂不可用」 |
| 学院 API 不可用 | 手动录入/H5 降级 | 明确提示，不显示假数据 |
| 多角度图片不可用 | 展示封面占位图 | 已有设计，保持 |
| OSS 不可用 | 只读；禁止上传 | 明确错误提示 |
| Redis 不可用 | 限流/缓存失效；写操作仍走 DB | 监控告警 |
| 搜索不可用 | 错误提示 | **禁止 mock 静默兜底** |

**实现：** `request.js` / `app.js` 按 build 模式设置 `useMock`；prod 构建增加 mock 检测门禁。

## 十九、风险与不做增强的后果

| 风险 | 不做增强的可能后果 |
|------|---------------------|
| 无自动化测试 | 每次发版回归靠人工，易引入报名/权限回归 bug |
| 无监控 | 故障依赖用户反馈，夜间/假期响应慢 |
| 无压测 | 开学/活动高峰可能出现慢响应或超卖争议 |
| 无审计 | 后台误操作无法追责 |
| mock 进生产 | 用户看到假数据，严重损害信任 |
| 无回滚/恢复演练 | 一次坏发布或数据事故导致长时间不可用 |
| 责任边界不清 | 与校方配合 Excel 冲突，交付扯皮 |
