## 一、项目概述与技术目标

### 1.1 项目背景

本项目为贵州交通职业大学中华文化书院「马院+书院」协同育人平台的数字化载体，以微信小程序（学生/访客端）+ PC 管理后台双端形态交付，围绕新闻资讯、线上展馆、文创展示、在线课程、资源下载、活动报名、学院矩阵互通、AI 智能服务等核心场景，实现中华优秀传统文化的线上传播与育人。

### 1.2 功能规模

平台共规划 11 个功能板块、移动端 65 项功能、管理后台 49 项功能，另含 11 项特色增值功能，共 125 项可验收功能，详见第八章覆盖矩阵。

### 1.3 技术目标

| 目标维度 | 指标 |
|---|---|
| 可用性 | 核心服务可用性 ≥ 99.5% |
| 性能 | 接口 P95 响应 < 300ms；缓存命中 < 50ms；首屏加载 < 2s |
| 并发 | 支撑数百用户同时在线浏览；音视频由 CDN 承载，不占用应用服务器带宽 |
| 数据准确 | 浏览量、观看量、DAU 统计准确、可追溯、可导出 |
| 可维护 | 管理员零技术背景可自主运营；代码模块化，便于后期扩展 |
| 成本 | 云资源与 AI 服务成本可控，整体在项目预算内 |

### 1.4 设计原则

遵循「极简架构、操作简化、易搭建、低维护」原则：采用成熟稳定的主流技术栈，不引入微服务、K8s、Elasticsearch 等重型组件；把复杂度集中在架构选型阶段一次性解决，日常运营保持简单。

## 二、总体架构设计

### 2.1 分层架构

```
┌─────────────────────────────────────────────────────────┐
│  客户端层   微信小程序（分包加载） │ PC 管理后台（Vue 3）  │
└───────────────┬─────────────────────────┬───────────────┘
                │  HTTPS                   │  HTTPS
┌───────────────▼─────────────────────────▼───────────────┐
│  接入层    Nginx 反向代理 + HTTPS 卸载 + 限流              │
└───────────────┬─────────────────────────────────────────┘
┌───────────────▼─────────────────────────────────────────┐
│  应用层    Spring Boot 3 RESTful API（无状态，可水平扩展）  │
└──────┬──────────┬──────────┬───────────────┬────────────┘
       │          │          │               │
┌──────▼───┐ ┌────▼────┐ ┌───▼──────┐ ┌─────▼──────────┐
│ Redis 7  │ │ MySQL 8  │ │ OSS+CDN  │ │  AI 能力层      │
│ 缓存/计数 │ │ 业务主库  │ │ 媒体资源  │ │  文本/语音/向量 │
└──────────┘ └─────────┘ └──────────┘ └────────────────┘
                                              │
┌─────────────────────────────────────────────▼──────────┐
│  第三方   微信开放平台（登录/订阅消息/关联小程序/内容安全） │
└────────────────────────────────────────────────────────┘
```

### 2.2 关键请求链路

**内容浏览（读，高频）**
```
小程序 → Nginx → 应用服务 → Redis 查缓存
  命中 → 直接返回（< 50ms）
  未命中 → 查 MySQL → 回写 Redis（TTL 10min）→ 返回
```

**音视频播放（大流量）**
```
小程序 <video> 组件 → CDN 边缘节点（带签名 URL，不经应用服务器）
```

**AI 文化问答（非流式，内容安全优先）**
```
小程序提问 → 应用服务 → 输入内容安全审核
  → 知识库检索（MySQL 余弦相似度）→ 构建问答上下文
  → 调用大模型 → 输出内容安全审核 → 返回完整答案
  → 写会话记录
```

### 2.3 小程序分包策略

微信小程序主包限制 2MB，总包限制 20MB，本项目按功能分包：

```
主包（必须）：登录、首页、个人中心、公共组件
  约 600—800KB，保证冷启动速度

分包 A（内容）：新闻、展馆、文创
分包 B（学习）：课程、资源
分包 C（互动）：活动报名、搜索、学院矩阵
分包 D（智能）：AI 助手、海报生成
```

分包在用户进入对应功能时按需加载，不影响首次打开速度。

## 三、技术栈选型与依据

### 3.1 选型总表

| 层次 | 选型 | 说明 |
|---|---|---|
| 小程序前端 | 微信原生（WXML/WXSS/JS）+ TDesign | 原生兼容性最好，TDesign 为腾讯官方组件库 |
| 管理后台 | Vue 3 + TypeScript + Element Plus + ECharts + Vite | 国内后台事实标准，ECharts 满足数据看板需求 |
| 后端框架 | Spring Boot 3 + MyBatis-Plus（Java 17） | 稳定、生态完善，与"以 CRUD 为主"的项目性质匹配 |
| 数据库 | MySQL 8.0（utf8mb4，utf8mb4_unicode_ci） | 字符集支持 emoji 与中文排序 |
| 缓存 | Redis 7 | 热点缓存、计数器、DAU 去重、限流 |
| 对象存储 + CDN | 阿里云 OSS + CDN（建议单一厂商）| 媒体文件统一存储，CDN 全国分发 |
| 图片处理 | OSS 图片处理（URL 参数按需缩放/格式转换）| 存原图一份，下发时按需生成 |
| 音视频转码 | 阿里云媒体处理 MPS / 云点播 VOD | 异步转码，上传完后台自动处理 |
| AI 文本 | 智谱 GLM-4.7-Flash（免费首选）+ 付费模型兜底 | 代码层抽象供应商，支持无改造切换 |
| AI 语音识别 | 阿里云录音文件识别 ASR（异步）| 视频字幕生成，一次生成长期复用 |
| 轻量 RAG | MySQL 存文本分段 + 应用层余弦相似度 | 适合本项目知识库规模，无需独立向量数据库 |
| 3D 展示 | 微信 xr-frame，降级为多角度图片轮播 | 无模型文件时降级，不影响页面结构 |
| 部署 | Docker + Docker Compose + 云服务器 | 环境一致，支持一键回滚 |

### 3.2 重点选型说明

**RAG 向量方案**
知识库规模为书院文化资料（通常数百到数千段），采用轻量方案：文本分段存入 `knowledge_chunk` 表，Embedding 向量以 JSON 存储，检索时在应用层计算余弦相似度取 Top-K。规模增大时可平滑迁移至 Qdrant，无需改动业务代码。

**中文全文搜索**
不引入 Elasticsearch。MySQL 8.0 配置 `ngram_token_size=2` 支持中文 FULLTEXT 索引，配合 `LIKE` 兜底。维护 `search_index` 表供跨模块搜索。

**接口版本控制**
所有接口带 `/v1/` 版本前缀：`/api/v1/news`。小程序无法强制用户升级，接口版本控制保障新旧版本共存。

## 四、媒体资源规范

### 4.1 图片规格

| 场景 | 推荐尺寸 | 比例 | 上传大小上限 | 格式 |
|---|---|---|---|---|
| 首页 Banner / 轮播图 | 1125 × 540px | 约 2:1 | 5MB | JPG |
| 新闻 / 课程 / 活动封面 | 750 × 422px | 16:9 | 3MB | JPG |
| 展馆详情大图 | 1500 × 1000px | 3:2 | 10MB | JPG |
| 文创工艺品图 | 1500 × 1500px | 1:1 | 10MB | JPG |
| 展馆 / 文创封面（卡片） | 750 × 750px | 1:1 | 3MB | JPG |
| 海报背景模板 | 750 × 1334px | 9:16 | 5MB | JPG/PNG |
| 图标 / Logo（透明背景）| 按实际 | — | 1MB | PNG |

**下发处理：** 图片统一存原图至 OSS，CDN 下发时通过 URL 参数按需缩放和转 WebP（节省 25—35% 流量）：
```
列表缩略图：?x-oss-process=image/resize,w_400/quality,q_80/format,webp
详情大图：  ?x-oss-process=image/resize,w_1200/quality,q_85/format,webp
用户放大：  原图 URL（不加参数）
```

### 4.2 视频规格

| 场景 | 输出分辨率 | 码率 | 格式 | 说明 |
|---|---|---|---|---|
| 课程教学视频 | 720p（1280×720）| 1—1.5 Mbps | MP4 H.264 | 手机屏幕下与 1080p 视觉差异小，文件更小 |
| 展馆 / 文创宣传视频 | 1080p（1920×1080）| 2—3 Mbps | MP4 H.264 | 视觉展示类，画质优先 |

管理员上传原始视频后，系统异步转码为标准版和高清版，并自动截取第 3 秒作为封面图，完成后更新课程记录状态。H.265 压缩率更高但低端机兼容性差，不采用。

### 4.3 音频规格

| 场景 | 格式 | 码率 | 上传大小上限 |
|---|---|---|---|
| 展馆语音讲解 | MP3 / AAC | 128kbps | 200MB |
| 课程配套音频 | MP3 / AAC | 128kbps | 200MB |

### 4.4 上传限制

| 类型 | 大小上限 | 格式白名单 |
|---|---|---|
| 图片 | 20MB | jpg / png / webp |
| 视频 | 2GB | mp4 / mov |
| 音频 | 200MB | mp3 / aac / m4a |
| 文档 | 100MB | pdf / doc / docx / ppt / pptx / xls / xlsx |
| 3D 模型 | 200MB | glb / gltf |

拒绝格式白名单以外的所有文件类型，文件名随机重命名存储，防止路径遍历。

**大文件上传策略：** 图片、视频、音频、3D 模型等大文件优先采用**前端直传 OSS/COS**（后端签发 STS 临时凭证，前端直接上传至对象存储），不经过应用服务器，避免占用服务器带宽和并发连接。服务器端 `client_max_body_size` 仅作后台小文件中转的兜底配置。

## 六、核心功能实现

### 6.1 内容浏览（读优化）

热点内容缓存至 Redis，TTL 10 分钟；列表分页结果缓存；内容更新时主动失效对应缓存（Cache-Aside 模式）。

### 6.2 音视频分发

所有视频/音频存入 OSS，CDN 全国节点分发，流量完全绕开应用服务器。CDN 下发带时效签名 URL（有效期 1—2 小时）防盗链。小程序 `<video>` 必须设置 `poster` 属性（封面图），避免加载期间黑屏。

### 6.3 图片懒加载

展馆详情页可能包含 10—20 张图片，小程序 `<image>` 组件开启 `lazy-load="{{true}}"`，滚动到视口时加载，避免进页面后同时请求大量图片。

### 6.4 浏览量统计

```
用户浏览 → Redis.INCR("view:{type}:{id}")
去重：Redis 记录 viewed:{userId}:{type}:{id}，TTL 30 分钟
定时任务（每 5 分钟）→ 批量增量同步至 MySQL
```

### 6.5 DAU 统计

```
用户首次访问 → Redis SADD dau:{date} {userId}
DAU 查询    → SCARD dau:{date}
凌晨定时任务 → 固化至 stat_daily，清理过期键
```

### 6.6 AI 文化问答（一期：非流式，内容安全优先）

一期采用非流式返回：先完成输入预审、知识库检索、模型生成和输出安全审核，全部通过后一次性返回完整答案。后续如需提升交互体验，可升级为流式输出模式（流式模式采用输入预审 + 输出过程拦截 + 完整答案复审的多层控制策略）。

**接口规范：**
```
POST /api/v1/ai/chat/sessions                    创建会话
POST /api/v1/ai/chat/sessions/{id}/messages      发送问题，返回完整答案
GET  /api/v1/ai/chat/sessions                    历史会话列表
GET  /api/v1/ai/chat/sessions/{id}/messages      会话消息记录
```

**请求体：** `{ "question": "阳明文化的核心是什么？" }`

**返回体：**
```json
{
  "answer": "……",
  "sources": [
    { "docTitle": "阳明文化资料", "chunkId": 123 }
  ]
}
```

**访问控制：** 未登录游客不可用；每用户每日上限 20 次；高频相同问题缓存答案（Redis，TTL 24 小时）。

**检索：** 将问题向量化，在 MySQL knowledge_chunk 表应用层计算余弦相似度，取 Top-5 相关段落拼入提示词。

### 6.7 AI 文案辅助

管理后台编辑器内置文案辅助功能，调用大模型对当前文本执行润色、扩写、摘要或标题生成，返回结果供编辑采纳，不自动替换原文。

**AI 文案生成规则：**

- 角色设定：书院文化内容编辑助手，文风典雅、庄重，适合面向师生与社会大众
- 支持操作：润色 / 扩写 / 生成摘要 / 生成标题 / 英文翻译
- 每次调用写入操作日志，每日调用次数设限

### 6.8 视频字幕生成

视频上传后，后台触发异步 ASR 转写任务。任务状态轮询（每 30 秒），完成后生成 `.vtt` 字幕文件存 OSS，关联课程记录。字幕一次生成、永久复用，费用与观看人数无关。

### 6.9 活动报名（数据库事务兜底）

名额控制以数据库事务作为最终一致性保障，Redis 仅用于缓存展示和削峰辅助：

```sql
-- 核心报名 SQL（原子操作）
UPDATE activity
SET enrolled_count = enrolled_count + 1
WHERE id = #{activityId}
  AND status = 'published'
  AND enrolled_count < quota;
-- 影响行数为 0 → 名额已满或活动不可报名
```

再插入 `enroll` 记录，`(member_id, activity_id)` 唯一索引防重复报名。

### 6.10 学院矩阵

按各学院实际条件选择呈现方式（后台按学院单独配置）：

| 方式 | 前提 | 推荐度 |
|---|---|---|
| 手动录入 | 对方提供图文素材 | ★★★★★ 最稳定 |
| H5 嵌入 | 对方有已备案 HTTPS 网页 | ★★★★ |
| 接口对接 | 对方提供开放 API | ★★★ 依赖对方 |
| 小程序跳转 | 双方在公众平台完成关联 | ★★★ 会出现系统确认弹窗 |

跳转时会出现微信系统确认弹窗，这是平台行为，无法绕过。`app.json` 中需在 `navigateToMiniProgramAppIdList` 预先声明目标 AppID（最多 10 个）。

### 6.11 订阅消息

微信订阅消息需用户主动授权（`wx.requestSubscribeMessage`），每次授权对应一次发送机会。发送逻辑：查 `member_subscribe_record` 表确认有可用次数 → 发送微信模板消息 → 扣减次数 → 同步写 `message` 表作为站内消息兜底，确保无论是否授权用户都能在消息中心看到通知。

## 七、安全与合规保障

### 7.1 接口安全

- 全站强制 HTTPS（TLS 1.2+）
- 接口统一 JWT 鉴权（`/api/v1/**` 路径），有效期 7 天
- 后台接口统一前缀 `/api/v1/admin/**`，单独拦截器校验管理员 Token
- RBAC 角色权限，最小权限原则；关键后台操作写 `sys_log`
- Nginx 层单 IP 限流（每秒请求上限），防 CC 攻击

### 7.2 内容安全（微信强制要求）

- 用户输入文本（AI 问答输入、反馈内容）经微信 `msgSecCheck` 审核
- AI 生成的完整答案经内容安全审核后再返回
- 后台发布流程建议设编辑—审核—发布环节（可配置为单人直接发布）

### 7.3 文件上传安全

- 格式白名单校验 + 文件大小校验（见第四章）
- 文件随机重命名存储，不保留原文件名
- OSS 使用 STS 临时凭证，不在客户端暴露永久密钥
- 图片经过安全检测（`imgSecCheck`）再存储

### 7.4 个人信息保护

- 手机号仅在报名时收集，首次登录不强制
- AI 会话不将手机号、身份证号等敏感字段送入模型
- 数据全程存储于国内云服务器，符合教育数据本地化要求
- 提供隐私政策与用户授权说明页（微信要求）

### 7.5 数据备份

- 数据库每日全量备份，保留 30 天
- OSS 文件多副本存储
- 每季度验证备份可恢复性

## 十、资源与成本估算

### 10.1 服务器三档方案

| 环境 | 部署方式 | 适用阶段 |
|---|---|---|
| 演示/测试 | 2 核 4G，同机运行 Nginx + Spring Boot + MySQL + Redis | 开发调试，不建议长期正式使用 |
| 正式低成本 | 应用服务器 2—4 核 4—8G + 云数据库 MySQL + Redis（同机或托管）+ OSS/CDN | 正式上线首选 |
| 稳妥生产 | 4 核 8G 应用服务器 + 高可用 RDS MySQL + 托管 Redis + OSS/CDN | 访问量增长后升级 |

正式环境建议采用应用服务与数据库分离部署。应用服务器运行 Nginx 与 Spring Boot 服务；业务数据存储在云数据库 MySQL；Redis 可根据预算选择同机部署或托管 Redis，访问量增长时可平滑拆分，无需改造代码。

建议所有云资源统一在一家厂商（阿里云或腾讯云），避免跨厂商流量费用和运维复杂度。

### 10.2 AI 服务成本

文本类 AI 能力优先采用免费或低价模型，适用于后台文案辅助和低频文化问答。字幕识别、向量化、视频转码等能力按实际用量计费，属低频或一次性成本。通过调用次数限制、答案缓存、额度预警和后台开关控制费用，整体在可控范围内。
