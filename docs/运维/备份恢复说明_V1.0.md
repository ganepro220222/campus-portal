## 1. 备份策略总览

| 对象 | 备份方式 | 频率 | 保留期 | RPO 目标 | 负责方 |
|------|----------|------|--------|----------|--------|
| MySQL 业务库 | RDS 自动全量 + 可选手动快照 | 每日 | 30 天 | ≤ 24h | 云运维 |
| Redis | 一般不单独备份（可重建缓存） | — | — | 浏览量 5min 落库 | — |
| OSS 媒体文件 | 多副本 + 可选跨区复制 | 实时 | 按桶策略 | 近实时 | 云运维 |
| 应用配置 | 环境变量台账 + `application-prod.example.yml` | 变更时 | 永久 | — | 运维 |
| 管理后台静态 | `admin/dist` 构建产物归档 | 每次发布 | 最近 5 版 | — | 运维 |
| 后端 JAR / 镜像 | Git Tag 对应构建物 | 每次发布 | 最近 5 版 | — | 运维 |
| 源代码 | Git `main` + Tag | 持续 | 永久 | — | 开发 |

**RTO 目标（建议）：** 数据库恢复 ≤ 2h；应用回滚 ≤ 30min（见 §5）。

## 2. 数据库备份

### 2.1 生产环境（RDS）

1. 在云控制台开启 **自动备份**，备份窗口建议凌晨低峰（如 02:00–04:00）。  
2. 设置保留 **30 天**（与技术方案 §7.5 一致）。  
3. 重大发布前执行 **手动快照**，命名：`pre-release-YYYYMMDD-vX.Y.Z`。  
4. 开启 **binlog**（默认），支持时间点恢复（PITR）。

### 2.2 本地开发（Docker Compose）

数据卷 `mysql_data` 持久化 MySQL 数据。开发机备份（可选）：

```bash
# 导出全库（在仓库根目录）
docker compose -f docker-compose.dev.yml exec -T mysql \
  mysqldump -uroot -pdev123456 --single-transaction --routines --triggers \
  --default-character-set=utf8mb4 shuyuan > backup/shuyuan_$(date +%Y%m%d).sql
```

恢复至空库：

```bash
# 警告：会覆盖当前 shuyuan 库数据
docker compose -f docker-compose.dev.yml exec -T mysql \
  mysql -uroot -pdev123456 --default-character-set=utf8mb4 shuyuan < backup/shuyuan_YYYYMMDD.sql
docker compose -f docker-compose.dev.yml restart backend
```

### 2.3 结构变更与 patch 脚本

| 类型 | 路径 | 说明 |
|------|------|------|
| 全量建表 | `sql/init.sql` | 新环境初始化 |
| 增量补丁 | `sql/patch-*.sql` | 须评审后按序执行 |
| 说明 | `sql/README.md` | patch 执行顺序与回滚注意 |

**原则：** 结构变更优先向前兼容；不可逆 DDL 须先有快照再执行。

## 3. Redis 说明

Redis 存储 **可重建** 的缓存与计数器：

| 键前缀 | 用途 | 丢失影响 |
|--------|------|----------|
| `login:*` | 登录失败锁定 | 用户可立即重试登录 |
| `rate:*` | 接口限流 | 计数重置 |
| `view:*` | 浏览量待落库增量 | 最多丢失 5 分钟内未落库增量 |
| `viewed:*` | 浏览去重 | 短时重复计数 |
| `dau:*` | 日活统计 | 当日 DAU 需重算 |
| `points:daily:*` | 积分每日上限 | 当日上限可能放宽 |

**浏览量兜底：** `ViewCountFlushJob` 每 5 分钟将 `view:*` 增量写入 MySQL `news.view_count`。

**生产建议：** 开启 Redis 持久化（AOF）或云托管高可用；**不以 Redis 作为唯一数据源**。

## 4. OSS 文件备份与恢复

### 4.1 备份策略

- OSS 默认 **多副本** 存储于同地域。  
- **P1 增强：** 开启跨区复制（CRR）至备用地域 Bucket。  
- 重要发布前可对 Bucket 做 **清单导出** 或增量同步至冷存储。

### 4.2 恢复场景

| 场景 | 处理 |
|------|------|
| 误删单个对象 | 版本控制开启时回滚至历史版本 |
| Bucket 策略误改 | 恢复 RAM 策略 + Bucket ACL/Policy |
| 区域性故障 | 从跨区复制 Bucket 切换 CDN 源站 |
| 数据库仍在、OSS 文件丢失 | 从备份 Bucket 按 `resource`/`course` 表 URL 映射回迁 |

### 4.3 配置恢复

OSS 连接信息通过环境变量配置（见 `docs/运维/环境变量说明.md` §2.5）。恢复后验证：

1. 后台上传测试图  
2. 小程序课程/展馆媒体可加载  
3. 私有 Bucket 直链返回 403，签名 URL 可访问

## 5. 应用与配置恢复

### 5.1 应用回滚（无数据库变更）

1. 停止当前 Spring Boot 进程 / 容器。  
2. 启动 **上一版本** JAR 或 Docker 镜像（与 Git Tag 对应，如 `v1.0.0-rc1`）。  
3. **环境变量保持不变**（`DB_*`、`REDIS_*`、`JWT_*` 等）。  
4. 管理后台：将 Nginx `admin/` 目录替换为上一版 `dist` 备份。  
5. 验证：`GET /api/v1/health` → `status=UP`，核心业务流程抽测。

详见 `docs/运维/部署手册_V1.0.md` §7。

### 5.2 配置恢复

| 配置 | 存放位置 | 恢复方式 |
|------|----------|----------|
| 后端环境变量 | 服务器 `/etc/shuyuan/env` 或 systemd `EnvironmentFile` | 从加密台账还原 |
| Nginx | `/etc/nginx/sites-available/shuyuan` | 配置库 / 备份文件 |
| 微信小程序 | 微信公众平台 | 域名、AppID 不变则无需恢复 |
| JWT Secret | 环境变量 | **丢失将导致全员登出**；须从台账恢复，不可随意更换 |

### 5.3 代码版本恢复

```bash
git fetch --tags
git checkout v1.0.0-rc1   # 示例 Tag
cd backend && mvn -B package -DskipTests
# 部署 target/*.jar
```

## 6. 数据库恢复演练（RDS）

**频率：** 首次上线前 ≥ 1 次；之后每季度 ≥ 1 次。

### 6.1 演练步骤

1. 在 RDS 控制台选择 **还原到备份** 或 **克隆实例** 至临时实例 `shuyuan-restore-test`。  
2. 将测试应用 `DB_URL` 指向临时实例（**勿指向生产**）。  
3. 启动应用，执行验证清单（§6.2）。  
4. 记录实际耗时 → **RTO**。  
5. 记录备份时间点与还原后最新数据时间差 → **RPO**。  
6. 销毁临时实例，恢复测试应用配置。

### 6.2 验证清单

- [ ] `GET /api/v1/health` → `db=UP`  
- [ ] 随机抽查 `news`、`hall`、`course` 记录数与生产一致（还原时点）  
- [ ] 管理员可登录后台  
- [ ] 小程序可拉取新闻列表  
- [ ] 报名数据 `enroll` 条数符合预期

## 7. 灾难恢复分级

| 级别 | 场景 | 优先动作 |
|------|------|----------|
| L1 | 单应用进程崩溃 | `systemd` / Docker 自动重启；查日志 `traceId` |
| L2 | 错误版本发布 | 应用回滚（§5.1），数据库不动 |
| L3 | 数据库误操作 | RDS PITR 或快照还原（§6） |
| L4 | OSS 部分文件损坏 | 版本回滚 / 跨区复制回迁（§4） |
| L5 | 全站不可用 | 按架构图逐层排查：DNS → Nginx → API → DB/Redis |
