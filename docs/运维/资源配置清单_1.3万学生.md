## 三、本校 1.3 万学生的流量模型（推导）

### 3.1 为什么不是「1.3 万并发」

- **13,000** = 潜在用户池，不是同时在线数。
- 云端书院属**非强制、低频文化类**应用，渗透率应低于选课/一卡通等刚需（见 §2.2 中青校媒、案例对比）。
- 验收与方案明确按 **数百同时在线、200 API 并发、50 报名并发** 设计，与 1.3 万总量级匹配。

### 3.2 三档场景假设（推导）

| 场景 | DAU（日活） | 占 13,000 比例 | 高峰同时在线 | 估算 API 并发 | 估算同时播放视频 |
|------|-------------|----------------|--------------|---------------|------------------|
| **保守**（上线初期） | 650 | 5% | 65–130 | 30–65 | 20–50 路 |
| **常态**（运营 1 学期后） | 1,300 | 10% | 130–260 | 65–130 | 50–150 路 |
| **峰值**（开学周/大型活动） | 2,600 | 20% | 400–520 | 150–200 | 150–300 路 |

**推导逻辑：**

1. **DAU 比例**：文化类非刚需，取 5%–20%（低于案例文中 3,200/12,458≈26%，因该案例含选课等高频功能）。
2. **高峰同时在线**：取 DAU 的 **10%–20%**（校园生活服务类调研常见「日打开 3–5 次」但分散全天；高峰集中在午间、晚间，见行业调研归纳）。
3. **API 并发**：同时在线用户中约 **50%** 在 3–5 秒内发起请求 → 在线 260 人时 API 并发约 **130**（与验收 200 留余量一致）。
4. **视频并发**：同时在线中约 **30%–60%** 可能处于课程/展馆页；其中部分播放视频 → 常态 **50–150 路**。

### 3.3 与验收指标的对照

| 验收指标 | 常态推导 | 峰值推导 | 是否匹配 |
|----------|----------|----------|----------|
| 200 并发浏览 | 65–130 | 150–200 | 常态有余，峰值需压测验证 |
| 50 并发报名 | 热门活动短时可达 | 活动推广日可达 | 必须做 50 并发脚本验证 |
| 数百同时在线 | 130–260 | 400–520 | 匹配 |

## 四、推荐资源配置（生产环境）

### 4.1 总原则（来自本项目 V1.2 §10.1）

1. **应用与数据库分离**（正式环境必须）。
2. **媒体走 OSS + CDN**，应用机 **不承担** 视频/课件流量。
3. **单机 4C8G 应用**为「稳妥生产」起点；访问量增长时**水平加应用机**，而非先换语言栈。
4. 云资源建议**同一厂商**（阿里云或腾讯云），避免跨云流量与运维复杂度。

### 4.2 推荐配置表（生产 prod）

| 组件 | 推荐规格 | 数量 | 依据 |
|------|----------|------|------|
| **应用服务器 ECS** | **4 vCPU / 8 GiB**（通用算力型，如 u2a/e） | 1 台 | V1.2 §10.1「稳妥生产」；§2.2 单机可支撑验收级 200 API 并发（缓存命中时） |
| **系统盘** | ESSD 40–80 GiB | 1 | 系统 + Nginx + Admin 静态资源 + 日志 |
| **公网带宽（应用机）** | **5 Mbps** 固定带宽，或按量 | — | 仅 API + Admin 页面，**不承载视频**；5 Mbps 足够约 **100+ 短 JSON QPS**（推导） |
| **云数据库 RDS MySQL** | **2 vCPU / 4 GiB**，存储 **100 GiB**，MySQL 8.0 | 1 实例 | 13k 用户 + 内容型业务，一年数据量通常 < 10 GiB（推导）；与开发 Docker 分离 |
| **云 Redis** | **1 GiB** 标准版（主从） | 1 实例 | DAU Set、公告缓存、积分日限、登录锁、限流；**256 MiB 活动价档过紧**，建议 1 GiB |
| **OSS 对象存储** | 标准存储，私有读，同地域 | 1 Bucket | 图片/视频/课件/字幕；直传 + 签名 URL |
| **CDN** | 按流量计费，中国大陆加速 | 绑定 OSS | 视频/图片/课件分发；**核心带宽支出项** |
| **Nginx** | 与应用同机部署 | — | HTTPS、反代 `/api/`、Admin 静态；配置见《详细开发实施方案 V1.2》§生产 Nginx |
| **域名 + SSL** | 1 个主域名 | — | 小程序服务器域名 + Admin HTTPS |
| **AI 能力** | 按量（文本大模型 + 内容安全） | 外部 API | 限次 20 次/人/日；**费用与 DAU 线性相关** |

### 4.3 预发 staging（建议）

| 组件 | 规格 | 说明 |
|------|------|------|
| ECS | **2 vCPU / 4 GiB** × 1 | 可应用+DB+Redis 小型混部，**仅 staging** |
| 数据 | 脱敏或子集 | 与 prod 配置结构一致，密钥隔离 |
| 用途 | V1.1 上线门禁「staging 跑通核心链路」 | 报名、登录、Admin CRUD、OSS 签名 |

### 4.4 开发 dev（现状）

| 组件 | 规格 | 说明 |
|------|------|------|
| 本机 / 2C4G | `docker-compose.dev.yml` | MySQL + Redis + Backend，**不用于正式评估容量** |

## 五、CDN 与存储容量估算（推导）

### 5.1 视频 CDN 峰值带宽

公式：`峰值带宽 ≈ 同时播放路数 × 平均码率`

| 场景 | 同时播放 | 码率取中值 | 峰值 CDN 带宽（推导） |
|------|----------|------------|------------------------|
| 常态 | 100 路 | 1.25 Mbps（720p 课程） | **≈ 125 Mbps（≈ 16 MB/s）** |
| 峰值 | 250 路 | 1.25 Mbps | **≈ 312 Mbps（≈ 39 MB/s）** |
| 峰值（含 1080p 宣传片） | 250 路混合 | 加权 ~1.8 Mbps | **≈ 450 Mbps** |

**结论：** 视频压力在 **CDN/OSS**，不在 4C8G 应用机。阿里云 CDN 按流量/带宽计费均可；初期用 **流量包** 更可控（见 §七）。

### 5.2 月流量估算（常态）

假设 **DAU 1,300**，其中 **30%** 当日观看视频，平均 **15 分钟/人**，码率 **1.25 Mbps**：

```
日视频流量 ≈ 1300 × 30% × 15 × 60 × 1.25 Mbps
           ≈ 390 人 × 900 秒 × 0.15625 MB/s
           ≈ 54.7 GB/天
月视频流量 ≈ 1.6 TB/月（仅视频）
```

再加上图片、课件下载、音频（按 V1.2 规格折算），**全站 CDN 月流量建议按 2–3 TB 规划**（常态推导）；上线初期可申请 **500 GB–1 TB 流量包**，按监控再扩容。

### 5.3 OSS 存储（首年）

| 内容 | 粗算 | 说明 |
|------|------|------|
| 课程视频 | 50 门 × 500 MB ≈ 25 GB | 校方素材规模可调 |
| 图片/文创/新闻 | 10–30 GB | WebP 缩略图经 CDN 参数 |
| 课件 PDF 等 | 10–50 GB | 资源下载模块 |
| 字幕/备份 | 5 GB | ASR 生成 VTT |
| **合计** | **50–110 GB** | 建议 Bucket 按 **200 GB** 额度规划 |

### 5.4 MySQL 存储

| 项 | 粗算 |
|----|------|
| 用户与业务行 | 13k 用户 + 内容表 + 报名/积分/日志 |
| 首年 | **< 10 GB**（推导） |
| RDS 购买 | **100 GB** 磁盘（与 V1.1「云 RDS」一致，留备份与日志空间） |

### 5.5 Redis 内存

| 用途 | 估算 |
|------|------|
| DAU Set `dau:yyyy-MM-dd` | 按日 1.3 万 memberId，约数十 MB/键 |
| 公告/列表缓存 | 10–50 MB |
| 积分日限、登录锁 | < 10 MB |
| 限流计数（若启用） | 10–30 MB |
| **建议** | **1 GiB** 规格（保留峰值活动日余量） |

## 六、应用层与并发能力对照

### 6.1 本配置能覆盖的并发（推论）

| 场景 | 目标 | 本配置预期 | 条件 |
|------|------|------------|------|
| API 浏览 | 200 并发，P95 < 500ms | **可满足** | 热点接口 Redis 缓存（公告已有；列表/详情按 V1.2 补齐） |
| 活动报名 | 50 并发不超卖 | **可满足** | 已有原子 `incrEnrolledCount`；**须压测报告** |
| 同时在线 | 数百 | **可满足** | 单台 4C8G |
| 视频播放 | 150–300 路 | **可满足** | 依赖 CDN，与 ECS 无关 |
| 课件下载 | 100+ 路 | **可满足** | CDN + 签名 URL |
| AI 同时对话 | 10–30 人 | **需限流** | 瓶颈在外部大模型 API 配额与延迟，非 CPU |

### 6.2 单机 4C8G 应用机 JVM 建议（推导，非强制）

```bash
# 8 GiB 机器，Spring Boot fat jar
-Xms2g -Xmx4g
-XX:+UseG1GC
```

| 进程 | 内存 |
|------|------|
| JVM 堆 | 2–4 GiB |
| Nginx + OS | 1–2 GiB |
| 余量 | 缓冲与监控 |

**Tomcat 线程（IO 密集）：** 参考 §2.2，`threads.max` 可由默认 200 调至 **300–500**，须 staging 压测后定稿（《详细开发实施方案 V1.2》亦要求上线后监控 CPU/堆）。

### 6.3 何时扩容（触发条件）

| 信号 | 动作 |
|------|------|
| 应用 CPU 持续 > 70%（高峰 10 分钟） | 先调优/加缓存，再加 **第 2 台 4C8G** 应用机 + SLB |
| MySQL CPU > 70% 或慢查询增多 | RDS 升至 4C8G 或读写分离（P1） |
| Redis 内存 > 80% | 升至 2 GiB 或清理 TTL |
| CDN 月流量 > 购买包 80% | 升级流量包或改带宽计费 |
| AI 费用超预算 | 收紧日限、加强答案缓存（V1.2 已有 24h TTL 设计） |

## 十、配置清单速查（采购/运维用）

### 10.1 生产环境 prod（推荐起步）

```yaml
# 逻辑清单，非云厂商 API 格式
prod:
  ecs_app:
    spec: 4vCPU_8GiB
    count: 1
    disk: 80GiB_ESSD
    bandwidth: 5Mbps
    roles: [nginx, spring_boot, admin_static]
  rds_mysql:
    spec: 2vCPU_4GiB
    storage: 100GiB
    version: "8.0"
  redis:
    spec: 1GiB_standard_ha
  oss:
    storage_class: standard
    acl: private
    cdn: enabled
  cdn:
    billing: traffic_package  # 建议首购 1TB，按监控扩容
  targets:
    api_concurrent: 200
    enroll_concurrent: 50
    peak_online_users: 500
    peak_video_streams: 300
```
