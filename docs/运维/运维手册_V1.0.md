## 1. 服务清单与职责

| 组件 | 职责 | 探活方式 |
|------|------|----------|
| 后端 API | 小程序 + 管理端接口 | `GET /api/v1/health` |
| MySQL | 业务数据 | health 中 `db` 字段 |
| Redis | 限流、浏览计数、登录锁 | health 中 `redis` 字段 |
| Nginx | HTTPS 终止、静态资源 | 云监控 / `curl -I https://域名` |
| OSS/CDN | 媒体文件 | 控制台 + 随机签名 URL 抽测 |
| 管理后台 | Vue 静态页 | 浏览器访问 `/admin/` |

**RPO / RTO 目标：** 见 [备份恢复说明_V1.0.md](./备份恢复说明_V1.0.md) §1（RPO ≤ 24h，RTO ≤ 4h）。

## 2. 日常巡检（建议每日 1 次）

### 2.1 自动化探活

```bash
curl -s http://<API_HOST>:8080/api/v1/health | jq .
```

期望：

```json
{
  "code": 200,
  "data": {
    "status": "UP",
    "db": "UP",
    "redis": "UP",
    "version": "x.y.z",
    "profile": "staging"
  }
}
```

| 字段 | 异常时动作 |
|------|------------|
| `status` ≠ UP | 立即执行 §4.1 |
| `db` = DOWN | §4.2 |
| `redis` = DOWN | §4.3 |

### 2.2 日志抽查

```bash
# Docker 示例
docker compose logs --tail=200 backend | grep -E "ERROR|WARN|traceId"
```

关注：

- 连续 `系统异常`（5xx）
- `登录锁定` 异常增多（可能撞库）
- OSS 上传失败
- AI 调用超时

### 2.3 资源与备份（云控制台）

| 检查项 | 阈值 | 处理 |
|--------|------|------|
| ECS CPU/内存/磁盘 | > 85% 持续 5min | 扩容或清理日志 |
| RDS 连接数 | > 80% max | 查慢 SQL / 连接泄漏 |
| RDS 自动备份 | 最近 24h 有成功记录 | 失败则 §5.3 |
| Redis 内存 | > 80% | 检查 key 暴涨、调整 maxmemory |

### 2.4 业务抽测

按 [dev回归自检清单.md](./dev回归自检清单.md) §3 抽 3–5 条核心链路（生产环境用体验版/正式版）。

## 3. 告警配置（E2-1）

### 3.1 应用内 Webhook 告警

后端已实现 **低成本默认告警**（钉钉 / 企业微信机器人 Webhook）：

| 告警类型 | 触发条件 | 冷却 |
|----------|----------|------|
| 健康检查异常 | `db` 或 `redis` 为 DOWN | 15 分钟 |
| 5xx 错误率 | 5 分钟窗口内请求 ≥ 20 且 5xx 占比 ≥ 1% | 15 分钟 |

**环境变量（staging/prod 建议配置）：**

| 变量 | 示例 | 说明 |
|------|------|------|
| `ALERT_ENABLED` | `true` | 开启告警任务 |
| `ALERT_WEBHOOK_URL` | `https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=...` | 机器人地址 |
| `ALERT_ERROR_RATE_PERCENT` | `1` | 错误率阈值（%） |
| `ALERT_MIN_SAMPLE_SIZE` | `20` | 最低样本量 |
| `ALERT_COOLDOWN_MINUTES` | `15` | 同类告警冷却 |

> dev 本地默认 `ALERT_ENABLED=false`，不影响开发。

### 3.2 云平台告警（必配）

| 告警 | 建议通道 |
|------|----------|
| ECS CPU/内存/磁盘 > 85% | 短信 + 企微 |
| RDS 备份失败 | 立即通知 |
| 5xx（Nginx access log） | 日志服务告警 |

### 3.3 告警演练（上线前 1 次）

1. staging 配置 `ALERT_WEBHOOK_URL` 指向测试群。  
2. 临时停 Redis 容器，等待 ≤ 5 分钟，确认收到「健康检查异常」。  
3. 恢复 Redis，记录演练时间与截图，附入测试报告。

## 4. 故障响应 Runbook

### 4.1 API 整体不可用（health status=DOWN）

```
1. 确认范围：仅 API / 含 Admin / 含小程序
2. curl /api/v1/health 看 db、redis 哪项 DOWN
3. 查看 backend 最近 50 行日志
4. 若刚发布 → 考虑 §6 回滚
5. 恢复后复测 health + 登录 + 新闻列表
```

### 4.2 数据库异常（db=DOWN）

```
1. RDS 控制台看实例状态、连接数、磁盘
2. 尝试只读账号 SELECT 1
3. 检查 DB_URL / 白名单 / 密码是否被轮换
4. 必要时从最近快照恢复（见备份恢复说明 §3）
```

### 4.3 Redis 异常（redis=DOWN）

```
1. 检查 Redis 进程 / 云实例状态
2. 重启 Redis（缓存可重建，浏览量最多丢 5min 未落库增量）
3. 重启 backend 清连接池
4. 观察 ViewCountFlushJob 是否正常落库
```

### 4.4 5xx 错误率告警

```
1. 查日志中最近 ERROR 堆栈与 traceId
2. 对照 Apifox 回归清单复现单接口
3. 常见原因：未执行 patch SQL、配置缺失、第三方超时
4. 修复后观察下一窗口错误率是否回落
```

### 4.5 小程序「服务器内部错误」

```
1. 开发者工具 Network 看具体路径与响应 code/message
2. 对照是否后端未重启（新路由 404/500）
3. 检查合法域名、HTTPS 证书
4. 参考 dev回归自检清单 §5
```

### 4.6 OSS / 媒体无法访问

```
1. 确认 OSS_ENABLED 与 AK/SK 有效
2. 私有桶须走签名 URL，禁止裸链
3. CDN 域名是否过期
4. 后台上传失败看后端日志 OSS 相关 WARN
```

## 5. 备份与恢复

### 5.1 日常备份

- **生产：** RDS 自动备份 + 重大发布前手动快照（命名 `pre-release-YYYYMMDD-vX.Y.Z`）  
- **本地：** 可选 `mysqldump` 见 [备份恢复说明_V1.0.md](./备份恢复说明_V1.0.md) §2.2

### 5.2 恢复流程（摘要）

1. 停止 backend 写入（维护页或停服）  
2. 从快照 / dump 恢复 MySQL  
3. 重启 backend，验证 health  
4. 抽测报名/新闻/登录  
5. 填写 RPO/RTO 实测记录

### 5.3 备份失败告警

RDS 控制台备份失败时：

1. 立即手动补快照  
2. 在告警群通报  
3. 应用侧可调用 `OpsAlertService.notifyBackupFailure(detail)`（供脚本集成，可选）

## 6. 发布与回滚

### 6.1 发布前检查

- [ ] CI 三路绿（backend test / admin build / mock guard）  
- [ ] 数据库 patch 已在 staging 执行  
- [ ] `ALERT_ENABLED` 与 Webhook 已配置  
- [ ] 备份已完成  

### 6.2 回滚（30 分钟内恢复服务）

```
1. 切换 Nginx / 容器到上一版本镜像或 JAR
2. Git checkout 对应 Tag（如 v1.0.0-rc1）
3. 若本次发布含不可逆 DDL，须先恢复 DB 快照再回滚代码
4. health UP 后通知业务方
```

详见 [部署手册_V1.0.md](./部署手册_V1.0.md) §9、[备份恢复说明_V1.0.md](./备份恢复说明_V1.0.md) §5。
