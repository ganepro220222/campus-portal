# 云端书院 · 部署手册 V1.0

> 贵州交通职业大学 ·「马院+书院」协同育人项目  
> 对照：《交付物与验收标准 V1.2》§一 交付物 §6、《工程化与准生产增强方案 V1.1》E4-2  
> 最后更新：2026-07-11

本文档说明如何在全新环境完成本地开发部署、预发（staging）与生产（prod）部署，以及健康检查、日志与回滚要点。

## 1. 架构概览

```
                    ┌─────────────┐
  微信小程序 ──────►│   Nginx     │◄──── 浏览器（管理后台）
                    │ 443 / HTTPS │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        /api/* →      /admin/*      静态 CDN
        Spring Boot   Vue 静态页     （OSS，可选）
        :8080
              │
        ┌─────┴─────┐
        ▼           ▼
     MySQL 8     Redis 7
```

| 组件 | 技术 | 说明 |
|------|------|------|
| 后端 API | Spring Boot 3 + JDK 17 | 统一前缀 `/api/v1` |
| 管理后台 | Vue 3 + Vite | 构建产物部署至 Nginx `/admin/` |
| 小程序 | 微信原生 | 请求 `baseUrl` 指向 API 域名 |
| 数据库 | MySQL 8.0 | 字符集 `utf8mb4` |
| 缓存 | Redis 7 | 登录锁、限流、浏览计数 |
| 媒体 | 阿里云 OSS（可选） | 大文件建议直传 OSS |

## 3. 本地开发部署（Docker）

### 3.1 启动

在仓库根目录执行：

```bash
docker compose -f docker-compose.dev.yml up -d --build
```

首次构建可能需数分钟。服务说明：

| 服务 | 端口 | 说明 |
|------|------|------|
| mysql | 3306 | 自动执行 `sql/init.sql` + `sql/seed-dev.sql` |
| redis | 6379 | 密码 `dev123456` |
| backend | 8080 | profile=`docker` |

### 3.2 健康检查

```bash
curl http://localhost:8080/api/v1/health
```

期望 `data.status` 为 `UP`，且 `db`、`redis` 均为 `UP`。

### 3.3 管理后台

```bash
cd admin
npm install
npm run dev
```

浏览器打开 http://localhost:5173 ，默认账号 `admin` / `Admin@123`。

Vite 开发服务器已将 `/api` 代理到 `http://localhost:8080`。

### 3.4 小程序

1. 微信开发者工具打开 `miniapp/` 目录  
2. 确认 `miniapp/config/env.js` 中 `dev.baseUrl` 为 `http://localhost:8080/api/v1`  
3. 开发环境微信登录走 `dev-mode`，无需真实 AppID  

**测试账号：** 学号 `2021001` / 密码 `Admin@123`

### 3.5 重置数据库

```bash
docker compose -f docker-compose.dev.yml down -v
docker compose -f docker-compose.dev.yml up -d --build
```

`-v` 会删除 `mysql_data` 卷，重新导入建表与种子数据。

### 3.6 常用运维命令

```bash
# 查看后端日志
docker compose -f docker-compose.dev.yml logs -f backend

# 进入 MySQL
docker compose -f docker-compose.dev.yml exec mysql mysql -uroot -pdev123456 shuyuan

# 清除登录锁定（开发环境）
docker compose -f docker-compose.dev.yml exec redis redis-cli -a dev123456 KEYS "login:*"
```

## 4. 数据库初始化

### 4.1 全新库

执行顺序：

1. `sql/init.sql` — 建表、索引、基础配置  
2. `sql/seed-dev.sql` — 仅**开发/演示**使用（测试账号、样例内容）

若版本升级带有 `sql/patch-*.sql`，按文件名日期顺序在目标库执行，并在变更记录中注明。

## 5. 预发 / 生产部署

### 5.1 部署前检查清单

对照 `docs/运维/环境变量说明.md` §6 逐项确认：

- [ ] `SPRING_PROFILES_ACTIVE` 为 `staging` 或 `prod`（非 `dev`）
- [ ] `JWT_SECRET` 已更换为 ≥32 位随机串
- [ ] `DB_PASSWORD` / `REDIS_PASSWORD` 非 `dev123456`
- [ ] `shuyuan.wx.dev-mode=false`
- [ ] 微信小程序 `WX_APPID` / `WX_SECRET` 已配置
- [ ] OSS 已配置（媒体主链）或已确认降级方案
- [ ] 无 `.env`、`application-prod.yml` 含真实密钥被提交到 Git
- [ ] 小程序 `useMock=false`，`baseUrl` 指向正式 API 域名

### 5.2 环境变量

复制根目录 `.env.example` 为部署机本地 `.env`，填入真实值。完整说明见 `docs/运维/环境变量说明.md`。

**staging / prod 必填：**

| 变量 | 说明 |
|------|------|
| `SPRING_PROFILES_ACTIVE` | `staging` 或 `prod` |
| `DB_URL` / `DB_USERNAME` / `DB_PASSWORD` | RDS 连接信息 |
| `REDIS_HOST` / `REDIS_PORT` / `REDIS_PASSWORD` | Redis 连接信息 |
| `JWT_SECRET` | JWT 签名密钥 |
| `WX_APPID` / `WX_SECRET` | 微信小程序凭证 |

**建议配置：**

| 变量 | 说明 |
|------|------|
| `OSS_ENABLED` | `true` 启用媒体上传 |
| `OSS_*` | Bucket、AK/SK、CDN 域名 |
| `AI_API_KEY` | 智谱 GLM Key；未配置时 AI 走知识库 fallback |
| `APP_VERSION` | 健康检查与发布标识 |

激活 profile：

```bash
export SPRING_PROFILES_ACTIVE=prod   # Linux
set SPRING_PROFILES_ACTIVE=prod    # Windows CMD
```

### 5.3 后端部署

**方式 A：Docker 镜像（推荐）**

```bash
cd backend
docker build -t shuyuan-backend:latest .
docker run -d --name shuyuan-api \
  --env-file /path/to/.env \
  -p 8080:8080 \
  --restart unless-stopped \
  shuyuan-backend:latest
```

**方式 B：JAR 直跑**

```bash
cd backend
./mvnw -DskipTests package
java -jar target/backend-0.0.1-SNAPSHOT.jar
```

需保证进程管理（systemd / supervisor）与环境变量注入。

### 5.4 管理后台构建

```bash
cd admin
npm ci
npm run typecheck
npm run build
```

将 `admin/dist/` 内容部署到服务器 `/var/www/admin/`（与下文 Nginx 配置对应）。

生产构建时可通过环境变量指定 API 地址（若前后端不同域）：

```bash
VITE_API_BASE_URL=https://your-domain.com/api/v1 npm run build
```

（当前默认走同源 `/api` 反代，多数场景无需设置。）

### 5.5 Nginx 配置

参考《详细开发实施方案 V1.2》Phase 8，核心片段如下：

```nginx
server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name your-domain.com;

    ssl_certificate     /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    client_max_body_size 2100m;

    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_buffering off;
        proxy_read_timeout 120s;
    }

    location /admin/ {
        root /var/www;
        try_files $uri $uri/ /admin/index.html;
    }
}
```

注意：`proxy_pass` **不要**在末尾加 `/`，否则 `/api/v1` 前缀会被剥离导致 404。

### 5.6 管理后台访问安全（建议）

本地开发用 `http://localhost:5173` 没有证书，这是正常现象。**上线后必须走 HTTPS**，验收清单也要求「所有接口强制 HTTPS」。

#### 管理后台要不要挂公网？

**不一定。** 常见分工如下：

| 访问对象 | 是否需要公网 | 说明 |
|----------|--------------|------|
| 师生小程序 | 需要 | 调用 `https://域名/api/v1/...` |
| 内容编辑人员 | 视学校网络而定 | 能打开管理后台、能调后端 API 即可 |
| 大模型服务 | 服务器出站 | Key 在后端，由服务器访问智谱等接口，**不经过编辑电脑直连** |

因此：**AI 文案辅助不要求管理后台对全网开放**。编辑在校内网或 VPN 里能访问后台，AI 功能就能用。

#### 推荐做法（由易到严）

1. **HTTPS 必开** — 用云厂商免费证书或 Let's Encrypt，在 Nginx 配置 `443`（见上文 5.5）。
2. **限制谁可以打开 `/admin/`** — 只允许学校办公网、机房出口 IP 或 VPN 网段访问（见下方 Nginx 示例）。
3. **首次登录改密码** — 禁用默认密码 `Admin@123`（见《管理员操作手册》）。
4. **按需开 AI Key** — 未配置 `AI_API_KEY` 时走演示兜底；正式环境在服务器环境变量中配置，**不要写进前端代码或 Git**。

#### Nginx：仅允许指定 IP 访问管理后台

将学校出口 IP 换成实际网段（可向信息中心索取）。未在白名单内的访问会返回 403：

```nginx
# 学校办公网 / VPN 出口（示例，请按实网修改）
geo $admin_allowed {
    default 0;
    10.0.0.0/8      1;   # 示例：校内网段
    203.0.113.50    1;   # 示例：固定办公出口 IP
}

server {
    listen 443 ssl;
    server_name your-domain.com;

    # ... ssl_certificate 等配置同 5.5 ...

    # 小程序与师生端：保持公网可访问
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 管理后台：建议仅白名单 IP 可访问
    location /admin/ {
        if ($admin_allowed = 0) {
            return 403;
        }
        root /var/www;
        try_files $uri $uri/ /admin/index.html;
    }
}
```

若编辑人员出差需临时访问，可短期追加 IP 到 `geo` 块，或改用学校统一 **VPN** 接入后再访问后台。

#### 已有防护（无需额外开发）

| 能力 | 作用 |
|------|------|
| 管理员登录 + 权限 | 未登录 401，越权 403 |
| 登录失败锁定 | 连续输错 5 次锁定 5 分钟 |
| AI 限流与安全审核 | 防刷接口、拦截违规内容 |
| 操作日志 | 后台写操作记入 `sys_log` |

**小结：** 小程序 API 对公网 + HTTPS；管理后台尽量 **内网 / VPN / IP 白名单**，不必为了 AI 而把后台裸露在无证书的公网环境。

### 5.7 小程序配置

修改 `miniapp/config/env.js`：

- `ENV` 设为 `staging` 或 `prod`
- `baseUrl` 设为 `https://your-domain.com/api/v1`
- 微信公众平台配置服务器域名（request 合法域名）

### 5.8 部署后验证

| 步骤 | 命令 / 操作 | 期望 |
|------|-------------|------|
| 健康检查 | `GET /api/v1/health` | `status=UP` |
| 管理端登录 | 浏览器访问 `/admin/` | 可登录并看到首页 |
| 接口文档 | staging 访问 `/swagger-ui.html` | 可打开（prod 默认关闭） |
| 小程序 | 真机微信登录 | 返回有效 token |
| 内容发布 | 后台发一条新闻 | 小程序端可见 |

## 6. 可观测与排障

### 6.1 健康检查

`GET /api/v1/health` 返回示例：

```json
{
  "code": 200,
  "data": {
    "status": "UP",
    "db": "UP",
    "redis": "UP",
    "version": "0.0.1-SNAPSHOT",
    "profile": "docker",
    "time": "2026-07-08T18:00:00"
  }
}
```

`db` 或 `redis` 为 `DOWN` 时，检查连接串、安全组、密码与白名单。

### 6.2 TraceId

- 请求可携带 `X-Trace-Id`，响应头原样回传  
- 后端日志含 `traceId` 字段，便于按单次请求排查  

### 6.3 日志位置

| 环境 | 位置 |
|------|------|
| Docker dev | `docker compose logs backend` |
| JAR / systemd | 标准输出或 `journalctl -u shuyuan-api` |
| Nginx | `/var/log/nginx/access.log`、`error.log` |

### 6.4 常见问题

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 502 / 连接拒绝 | 后端未启动或端口不对 | 检查 8080 进程与健康接口 |
| 401 请先登录 | token 过期或未传 Authorization | 重新登录 |
| 429 操作过于频繁 | 触发限流（登录/报名/AI） | 等待窗口结束或调整 `shuyuan.rate-limit` |
| 上传失败 | OSS 未配置或超限 | 检查 `OSS_*` 与 `OSS_MAX_UPLOAD_BYTES` |
| 微信登录失败 | `dev-mode` 未关或 AppID 错误 | 核对 `WX_APPID`/`WX_SECRET` |

## 7. 回滚

### 7.1 应用回滚

1. 保留每次发布的 JAR / 镜像 tag（建议与 Git tag 对应，如 `v1.0.0`）  
2. 停止当前容器或进程  
3. 启动上一版本镜像 / JAR，**环境变量保持不变**  
4. 验证 `/api/v1/health` 与核心业务流程  

### 7.2 数据库回滚

- 优先使用 RDS 时间点恢复或发布前手动快照  
- 慎用不可逆 DDL；结构变更应通过 `patch-*.sql` 评审后执行  

### 7.3 管理后台回滚

替换 `/var/www/admin/` 为上一版 `dist` 备份，刷新 CDN（如有）。
